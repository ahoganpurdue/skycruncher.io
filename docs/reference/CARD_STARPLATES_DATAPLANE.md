# Starplates Dataplane — algorithm card (HEALPix cells, PM propagation, Arrow contract lane)

This card documents the Rust-side star-catalog dataplane (`src-tauri/src/starplates/`): how Gaia rows are bucketed into HEALPix cells, how a query cone finds the right cells, how proper motion is propagated to the observation date, the exact byte contract for the Arrow files that carry the data, and the content-addressed manifest that makes corruption and partial coverage honestly detectable rather than silently wrong. It is one of two R2-backed star-data systems in this codebase — the other is the quad-index/atlas-sector fetch pipeline described in `docs/R2_STARDATA_LAYOUT.md`; this card covers only the HEALPix-cell dataplane, not that one.

Symbols: (α,δ)=RA/Dec, **DEGREES everywhere in this lane** (the inverse of the engine's RA-in-HOURS house convention — see UNIT WARNINGS); order=HEALPix order, nside=$2^{order}$; $\mu_{\alpha^*}$=`pm_ra_masyr` (Gaia PM in RA, already ×cosδ), mas/yr; Δyr=Julian years since the shipped J2016.0 constant; sha=SHA-256 lowercase hex.
Every entry below is VERIFIED against the code at the cited line; formulas transcribed from the running code, not the literature.

### 1. HEALPix NESTED cell from Gaia `source_id` bit shift — zero `ang2pix` anywhere
Assigns every Gaia star to a HEALPix sky cell without any trigonometry at all, by reading the cell index directly out of the high bits of the Gaia `source_id` — a shortcut that only works because ESA encoded that lattice position into the ID at catalog-build time.
- **Formula**: Gaia DR3 `source_id = healpix12·2^35 + counter` (L12 NESTED lattice), so the order-N cell is a pure shift: $cell = source\_id \gg (59 - 2\cdot order)$ — order 5 ⇒ `>> 49`, order 6 ⇒ `>> 47`. $npix(order) = 12\cdot4^{order}$ (coded as `12u64 << (2·order)`). Conservative cell circumradius bound: $1.2\cdot\sqrt{\pi/3}/nside$ radians→degrees (≈2.199° at order 5) — the bare $\theta_{pix}=\sqrt{\pi/3}/nside$ carries an explicit 1.2× honesty margin in code (measured true circumradii ≤ ~0.9·θ_pix).
- **Code**: `src-tauri/src/starplates/geometry.rs:34-37` (`cell_of_source_id`, `debug_assert!(order <= 12)`), `:27-29` (`npix`), `:44-47` (`max_cell_radius_deg`); build-time BigInt twin `tools/starplates/build_release.mjs:104-110` (`cell5OfSourceId`, `sourceId >> 49n` + range throw), `:113-119` (`cell6OfSourceId`, `>> 47n`); spec `docs/STARPLATES_SPEC.md` §0.5 (encoding) / §2 (layout) / §3.1 (shift-only rule).
- **Cite**: Górski, Hivon, Banday, Wandelt, Hansen, Reinecke & Bartelmann (2005), "HEALPix: A Framework for High-Resolution Discretization and Fast Analysis of Data Distributed on the Sphere", ApJ 622, 759 (NESTED scheme; cited in-code at `geometry.rs:49`). The `source_id` bit layout is from the ESA Gaia DR3 archive data-model documentation (`gaia_source.source_id`) [VERIFY — archive documentation, not a peer-reviewed paper; the release papers describe content, not the bit layout].
- **Gotcha**: the shift yields a NESTED index only because ESA allocated `source_id` on the L12 NESTED lattice — it is meaningless for any non-Gaia identifier. Order bound is `debug_assert!` only (release builds skip it; `manifest.rs:93-99` enforces order ≤ 12 and `cell < npix(order)` upstream). In JS the shift MUST be BigInt (`>> 49n`): `source_id` exceeds `Number.MAX_SAFE_INTEGER` and a Number shift silently truncates (§3.1 tooling trap).

### 2. Cone→cells resolution: manifest data-derived bounds, unit-vector scan
Given a query point and radius on the sky, finds every HEALPix cell that could contain a matching star, using per-cell center/radius bounds that were measured directly from the actual data at build time rather than computed from idealized cell geometry.
- **Formula**: candidate cell ⇔ $\operatorname{angdist}(\hat q, \hat c_{cell}) \le radius_{query} + radius_{cell}$, where $\hat q,\hat c$ are unit vectors $[\cos\delta\cos\alpha, \cos\delta\sin\alpha, \sin\delta]$ and $\operatorname{angdist} = \arccos(\operatorname{clamp}(\hat a\cdot\hat b, -1, 1))$ — RA-wrap/pole safe by construction. Per-cell center/radius are DATA-derived at forge time: center = normalized mean member unit vector; radius = max member angular distance measured against the angle-QUANTIZED center (the exact manifest values), so every member provably sits inside the bound. Row filter after PM propagation: $\hat v\cdot\hat q \ge \cos(radius_{query})$.
- **Code**: `src-tauri/src/starplates/geometry.rs:13-18` (`unit_vec`), `:21-24` (`angdist_deg`); selection scan `src-tauri/src/starplates/query.rs:100-105`; true-cone row filter `:87-90`; bounds precomputed from the manifest at `src-tauri/src/starplates/store.rs:268-276`; forge-side bound derivation `tools/starplates/build_release.mjs:307-337` (`coneBound`).
- **Honesty counter (documented deviation)**: runtime HEALPix math exists ONLY for `cells_absent_release` — absent cells have no data-derived bounds, so `query.rs:141-154` scans ALL `npix(order)` geometric NESTED centers (`geometry.rs:69-111`, reference `pix2ang_nest` restricted to centers) against `radius_query + max_cell_radius_deg(order)`; `None` ⇒ honest `"--"` when the release has no T1 cells. Never used for data-cell selection (spec §15.B4, ruling §12.12).
- **Cite**: in-house design (spec §5.3.1) — the dot-product angular distance is elementary spherical geometry with no single canonical paper; the `nest_center_deg` cell-center function follows the Górski et al. (2005) `pix2ang_nest` reference algorithm (entry 1 cite).
- **Gotcha**: selection is a LINEAR scan over the manifest's T1 cells (10,612 populated at order 5) and the honesty counter over all 12,288 — no spatial index; fine at this scale, revisit at order 6 (49,152). The conservative circumradius can OVER-count `cells_absent_release` near coverage holes and trigger one extra T0 read — deliberate (never under-reports). Spec §2.3's example "cell 2417 ≈ (170.03, 12.45)" is illustrative ONLY: cell 2417's true center is (262.5°, +45.0°); the real M66 order-5 cells are 7041/7043/7044/7046, pinned with release data in `geometry.rs:174-197` (worst geometric-vs-data center distance over all 10,612 cells: 0.569°, cell 9894).

### 3. Proper-motion epoch propagation — v1-bit-identical, quirks preserved
Moves each star's stored Gaia position forward in time to the observation date using a simple first-order proper-motion model — deliberately the same simplified formula (with the same known quirks) an earlier version of this dataplane used, so results stay bit-identical across versions.
- **Formula** (exact code order): $pm_{ra,deg} = pm_{ra,masyr}/3.6\times10^6$; $pm_{dec,deg} = pm_{dec,masyr}/3.6\times10^6$ (mas/yr → deg/yr, f32 args widened to f64); **dec FIRST**: $dec' = dec + pm_{dec,deg}\cdot\Delta yr$; $\cos\delta' = \max(|\cos(dec')|, 0.001)$; $ra' = ra + (pm_{ra,deg}\cdot\Delta yr)/\cos\delta'$ ($\mu_{\alpha^*}$ is already ×cosδ, hence the division); return $((ra'+360) \bmod 360,\ dec')$. $\Delta yr = (epoch_{jd} - GAIA\_DR3\_EPOCH\_JD)/365.25$ with $GAIA\_DR3\_EPOCH\_JD = 2457388.5$ (`mod.rs:25`).
- **Code**: `src-tauri/src/starplates/query.rs:53-60` (`propagate_v1`; normative doc comment `:47-52`), Δyr at `:85`; hand-computed + v1 bit-parity + polar-clamp tests in the same file (`:226-282`). Constant identical to v1 `catalog::mmap_manager::GAIA_DR3_EPOCH_JD` (`mmap_manager.rs:6`) by spec §5.3 (normative formula block, spec lines 415-419).
- **Cite**: rigorous epoch propagation (full spherical treatment incl. radial velocity / perspective acceleration) is Butkevich & Lindegren (2014), "Rigorous treatment of barycentric stellar motion", A&A 570, A62 — deliberately NOT implemented; the code ships the first-order flat form because bit-parity with the earlier version is the contract (measured ≤2 mas against it over the pinned field, spec §15). $\mu_{\alpha^*}\equiv\mu_\alpha\cos\delta$ convention per ESA Gaia archive documentation [VERIFY — archive documentation, not peer-reviewed].
- **Gotcha**: (a) the shipped "J2016.0" constant 2457388.5 is 2016-01-01T00:00 = Julian epoch J2015.99863; textbook J2016.0 is JD **2457389.0** — a 12 h drift preserved for bit-parity with the earlier version, worth ≤0.00137·μ mas (~1.4 mas at μ=1000 mas/yr), inside the 2 mas parity bound but real doc-vs-textbook drift. (b) the 0.001 clamp caps $1/\cos\delta'$ at 1000, so RA motion is silently wrong for $|dec'| \gtrsim 89.943°$ — a preserved quirk, not a new bug. (c) dec′ is NOT clamped to ±90 (extreme μ·Δt can exceed the pole). (d) no radial-velocity/perspective terms anywhere. This is the Gaia-epoch-aware proper-motion path in this codebase — contrast with CARD_TIME_EPHEMERIS.md's generic (and, in at least one code path, seemingly J2000-anchored) precession/PM machinery, which is a different lane entirely.

### 4. Frozen Arrow IPC byte contract — cells (file) vs responses (stream)
Defines the exact binary layout every star-data file and network response on this dataplane must follow — column order, types, sort order, no null values — so that both the Rust writer and the TypeScript reader agree on every byte without needing a schema negotiation.
- **Contract**: 7 non-nullable columns in EXACT order — `ra_deg:f64, dec_deg:f64, pm_ra_masyr:f32, pm_dec_masyr:f32, g_mag:f32, bp_rp:f32, source_id:u64`; exactly ONE record batch; no validity buffers (per-column `null_count == 0` enforced on read); rows sorted `g_mag` asc (f32 `total_cmp`), ties `source_id` asc. Cells = Arrow IPC **file** format (`ARROW1` magic at both ends, footer); responses = Arrow IPC **stream** format (0xFFFFFFFF continuation marker … EOS). Rust writer: 64-byte alignment, `MetadataVersion::V5`.
- **Code**: `src-tauri/src/starplates/arrow_io.rs:46-56` (schema), `:58-62` (write options), `:65-71` (`sort_rows`), `:115-134`/`:190-208` (cell write/read incl. magic + exactly-1-batch + schema-by-POSITION checks), `:136-187` (type + null checks), `:229-260` (response stream + discrete provenance metadata keys `skycruncher.release/.epoch_jd/.positions/.tier_depth/.cells_served/.cell_shas/.cells_absent_release/.cells_absent_local` — verbatim wire literals, spec §5.2). TS consumer: `src/engine/pipeline/m6_plate_solve/starplates_provider.ts:47-55` (expected column table), `:116-156` (`decodeStarplatesResponse` — zero-copy `child.data[0].values` TypedArray handles; loud throws on >1 batch, missing column, or wrong physical type; `BigUint64Array` for `source_id`).
- **Cite**: Apache Arrow Columnar Format + IPC specification (arrow.apache.org/docs/format/Columnar.html) [VERIFY — project specification, not peer-reviewed].
- **Gotcha**: **folded-metadata determinism deviation** (`arrow_io.rs:13-21`, spec §15.B1): arrow-rs (=54.3.1) serializes schema metadata via `std::HashMap` iteration — per-process-random order — so Rust-written CELL files carry ONE metadata key whose value is the six logical fields key-sorted and `;`-joined (`cell_metadata_value`, `:96-111`); responses keep discrete keys (ephemeral, determinism not required; readers never depend on cell metadata — the manifest is authoritative). TS validates columns by NAME (`getChild`), Rust by position — TS would accept a column-reordered stream that Rust rejects (both reject wrong types). `epoch_jd` metadata is Rust shortest-round-trip `Display` ("as passed", e.g. `2461234.5`). **BigInt tooling trap** (§3.1 normative): `source_id` stays BigInt end-to-end in TS; the only stringification is `` `Gaia_${id}` `` (`:191`) — an earlier version printed the u32-TRUNCATED id; this dataplane carries the full u64.

### 5. Manifest SHA-256 content addressing + coverage-honesty counters
Every blob this dataplane serves is content-addressed and verified: a manifest pins a SHA-256 hash for every file, and the manifest itself is pinned by its own hash, so corruption anywhere in the chain is caught rather than silently served — and gaps in coverage are reported as honest counters instead of pretended away.
- **Formula**: every blob entry carries `sha256` = 64-LOWERCASE-hex SHA-256 of the blob bytes, enforced by `len==64 && all bytes ∈ [0-9a-f]` (uppercase rejected); the store's single mutable pointer `pinned.json {release, manifest_sha256}` pins the manifest's own SHA — `open()` re-hashes the manifest bytes and refuses on mismatch. Verified reads (§6.3 ladder): `ARROW1` magic on EVERY open; full SHA-256 on first open per session; failure ⇒ quarantine + treat locally absent — corruption never fails a solve. Counters: `cells_absent_release` (cone cells the release never had; `None` ⇒ `"--"`) and `cells_absent_local` (in release, not yet synced/corrupt).
- **Code**: `src-tauri/src/starplates/manifest.rs:46-48` (`is_hex64`), `:58-111` (`validate`: `format_version==1`, path-traversal rejection `..`/leading `/`/`\`/`:`, t1/t2 blobs MUST carry `healpix_order/cell/center/radius` — the cone bounds of entry 2, ≤1 t0 blob); `src-tauri/src/starplates/store.rs:214-242` (pin verification), `:350-377` (`read_blob_verified` + per-session verified-set), `:380-394` (quarantine); hasher `src-tauri/src/starplates/mod.rs:38-47`. Warning rule: `src/engine/pipeline/m6_plate_solve/starplates_provider.ts:165-173` (`partialCoverageWarning` — any counter > 0 MUST surface; emitted via `console.warn` at `:241-242`; Rust twin `query.rs:201-206`). Absent = absent, solve continues.
- **Cite**: SHA-256 per NIST FIPS 180-4 (algorithm standard); the in-code test vectors at `mod.rs:54-64` are the canonical NIST "abc" example plus the empty-input digest (NIST-published examples/CAVP vectors — the FIPS 180-4 document itself prints no digests) — a federal standard, not peer-reviewed literature; content-addressed immutable-blob storage is engineering practice (cf. git's object store) [VERIFY — no canonical peer-reviewed source].
- **Gotcha**: the honest-or-null decode means a response that omitted the counter keys would map to `null`, which `partialCoverageWarning` treats as 0 — the rule holds only because the Rust writer ALWAYS stamps both keys (`arrow_io.rs:243-250`). Quarantine is best-effort (failed rename ⇒ still absent this session, retried next). `cells_served` may carry a literal `t0` token after the ascending `order:cell` tokens whenever the T0 bootstrap blob was consulted (documented deviation, spec §15.B3) — parsers must not assume every token is `order:cell`.

### 6. Forge build determinism — full-u64 ingestion, BigInt partitioning, byte gates
Builds the release files themselves from a raw Gaia extract: partitions rows into HEALPix cells by the same bit-shift as entry 1, sorts and deduplicates them, splits off a bright-star prefix for fast shallow queries, and verifies the whole build is byte-for-byte reproducible before it ships.
- **Idea**: `gaia_vanguard_dr3.csv` (or chunk-dir; column-NAME-driven, extras ignored) → per-cell Arrow files partitioned SOLELY by the BigInt shift (entry 1) — never the older `vanguard.bin` format, whose stored `source_id` is u32-TRUNCATED with the HEALPix bits already gone (spec §0.6, ruling §12 row 11). Total row order `g_mag`(f32) asc, ties `source_id` asc; per-cell dedupe by `source_id` keep-first; T0 = the bright prefix ($G \le 9.0$ on the STORED f32, `Math.fround(t0MagMax)`) of each sorted T1 cell, so T0 is exactly a row-prefix (the LOD property). Determinism: no wall-clock/randomness/absolute paths in any output byte; key-sorted metadata; manifest floats ECMAScript shortest-round-trip with `f32Shortest` for f32-derived stats.
- **Code**: `tools/starplates/build_release.mjs:93` (`REQUIRED_COLUMNS`), `:188`/`:237-238` (`^[0-9]+$` + `BigInt(...)` — full u64, fails loudly on out-of-range cell), `:104-110` (shift partitioning), `:83-84` (`T0_MAG_MAX_DEFAULT=9.0`, `T1_MAG_MAX_NOMINAL=12.5`), `:80` (`HEALPIX_ORDER_T1=5`), `:427-433` (total-order comparator), `:439-452` (dedupe, duplicates counted never written), `:417-423` (boundary-cell exclusion), `:471-481` (f32 bright-prefix split), `:144-152` (`f32Shortest`), `:610-637` (`verifyDeterminism` — builds twice, byte-compares the manifest, re-hashes every blob against the first manifest); vitest gate `src/engine/tests/starplates_build_release.test.ts:222-257` (blob-sha + determinism assertions). Deviations ledger: `docs/STARPLATES_SPEC.md` §13 (D1-D7; D5 = test-only reference `ang2pix`, D6 = chunk-dir extraction lineage, D7 = dedupe/boundary/T0-metadata clarifications).
- **Cite**: Gaia DR3 catalog content: Gaia Collaboration, Vallenari et al. (2023), A&A 674, A1. Reproducible-build discipline per reproducible-builds.org [VERIFY — engineering practice, no peer-reviewed canonical source].
- **Gotcha**: "T1 = G ≤ 12.5" is NOMINAL — `T1_MAG_MAX_NOMINAL` is written into manifest prose only; the build path applies NO magnitude cut to T1 (every ingested row lands in its cell; the 12.5 bound comes from the upstream extraction query). The HIGHEST populated cell is excluded by default (`--no-exclude-boundary` to keep): the ESA TAP 3M-row cap is `source_id`-ordered, so the last cell is truncated mid-cell — its rows appear in stdout stats only, never the manifest. Node-forged cells are 8-byte aligned (apache-arrow JS, deviation D2) vs Rust 64-byte — both valid Arrow IPC, but the two writers are NOT byte-interchangeable for identical rows; determinism gates hold per-writer only.

## UNIT WARNINGS
- RA is **DEGREES** everywhere in this lane (wire columns, manifest bounds, geometry, store) — the OPPOSITE of the engine's RA-in-HOURS convention. Conversion happens ONLY at the adapter boundary: `queryStarsV2` sends `raCenterHours * 15.0` (`starplates_provider.ts:239`) and `toStandardStars` returns `ra_hours = ra_deg / 15.0` (`:194`). An hours/degrees slip here is silent.
- `pm_ra_masyr` IS Gaia $\mu_{\alpha^*}$ (already ×cosδ), mas/yr, f32 on the wire (~$10^{-4}$ mas quantization at catalog values); mas/yr → deg/yr is $/3.6\times10^6$, and propagation divides by $\cos(dec')$ — feeding plain $\mu_\alpha$ doubles-in the cosδ.
- Epochs are Julian Dates; Δt in Julian years (365.25 d). The shipped J2016.0 constant `2457388.5` is 12 h earlier than textbook J2016.0 (JD 2457389.0) — parity-preserved with the earlier dataplane version, see entry 3.
- Magnitude thresholds compare against the STORED f32 (`Math.fround`), never the CSV f64 — the T0/T1 prefix property depends on it.
- `source_id` is u64 > `Number.MAX_SAFE_INTEGER`: BigInt end-to-end in JS/TS (`>> 49n`, `BigUint64Array`); routing it through a JS Number truncates silently.
- `geometry.rs` takes and returns degrees; radians exist only transiently via `to_radians()`/`to_degrees()`. Cell-radius bounds are degrees.
