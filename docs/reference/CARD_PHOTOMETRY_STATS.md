# Photometry & Statistics — algorithm card (measurement & confirmation lane)

This card covers how the pipeline decides a blob of pixels is a star, measures its shape and brightness, and — after a solve — independently confirms the solution by measuring flux at exactly the positions the catalog predicts and comparing that against a scrambled-position control. It is the statistical backbone of "did we actually solve this field," separate from the solve itself.

Symbols: σ=standard deviation, k·σ = sigma multiple, z = standard-normal deviate, N/n = sample counts. All entries VERIFIED against the cited line; formulas transcribed from the running code, not idealized.

> **Cold-path scoping (legacy vs. current reference engine):** entries 4 (forced photometry at catalog positions) and 5 (set-level confirmation gate) describe `deep_verify.ts` / `forced_confirm.ts`, post-solve stages wired into `orchestrator_session` step 5 that consume the solver's matched-star output and WCS. The shipping *solve* for the desktop app is now the separate Rust reference engine by default (`VITE_SOLVER_GREENFIELD`, default ON — see `docs/SURFACE_CONVERGENCE.md`), but these two post-solve stages are TS/WASM-lane machinery that has not been ported to (or wired after) that engine's seam — its response is mapped into the same `PlateSolution` shape, and the shared TS pipeline's later steps, including these, still run against it in principle, but `deep_verify.ts`/`forced_confirm.ts` were built and last verified against the legacy engine's receipt shape and are unconfirmed against reference-engine-produced solutions. They remain fully live and accurate for the legacy solve path, which is still what browser, headless/API, and batch consumers exercise exclusively (none of them reach the Tauri-gated reference-engine seam). `SOLVER_CONFIRM_SET_EXCESS_Z` (entry 5) is a `SOLVER_*`-family env constant in `pipeline_config.ts`, flagged for eventual consolidation into the reference engine's own config surface if/when this machinery is ported — see `docs/local/GREENFIELD_SOLVER_CORE_BRIEF_2026-07-20.md`. Entries 1-3 (sigma-threshold detection, PSF fit, Richardson-Lucy) are detection/PSF-lane and unaffected — independent of which engine produced the solve.

### 1. Sigma-threshold detection + culling-reason tally
Finds candidate stars by thresholding the image against its own background statistics, then runs each candidate blob through a cascade of independently-tuned shape/brightness gates (not one global threshold) that route it to acceptance, rejection, or a specialized bucket (planet, hot pixel, saturated-star void).
- **Formula**: coarse mask threshold = mean + k·σ (global stats over sampled luminance), k=`sigFactor`=2.0. Downstream: an independently-tuned cascade of per-blob SNR/circularity/flux gates (LOW_SNR if bg≈black-floor & snr<15; circularity<0.8 reject only if snr<75; saturated-star void carve if circ>0.85 & snr>100; planet routing if snr>1200 or ephemeris-locked; hot-pixel anomaly if snr>300) — NOT one global threshold.
- **Code**: `src/engine/pipeline/m4_signal_detect/signal_processor.ts:49-53` (coarse mean+kσ), `:512-550` (per-reason cuts), `:654,727,739,750,768` (SNR/flux gates); tally reconciliation in `culling_stats.ts:32-63` (`computeCullingCounts` — never invents a "dropped" count when no pre-tally exists).
- **Cite**: generic sigma-clipped background thresholding; the canonical astronomical implementation of this pattern is SExtractor (Bertin & Arnouts 1996, A&AS 117, 393), but this is an independent from-scratch cascade, not a SExtractor port or reimplementation.
- **Gotcha**: the file's own header docstring claims "Thresholding (**5**-sigma)" (`:12`) — the LIVE coarse constant is **k=2.0σ** (`:51`). The 5σ figure is stale/aspirational documentation, not the shipped value; real star/junk separation happens in the downstream per-reason cuts, not the coarse mask.

### 2. Per-star PSF fit — Levenberg-Marquardt, 2D Gaussian
Fits a 2D Gaussian to each detected star's pixel stamp to get a sub-pixel center, width, and orientation — the basis for FWHM, ellipticity, and everything the PSF-characterization lane reports.
- **Model**: $F(x,y) = A\cdot\exp(-(a\,dx^2+b\,dx\,dy+c\,dy^2))$, $dx=x-cx$, $dy=y-cy$; a,b,c = standard rotated-Gaussian coefficients from $(\sigma_x,\sigma_y,\theta)$. Update: $(J^TJ+\lambda I)\delta=J^Tr$; $\lambda\times10$ on singular solve, $\lambda\times0.1$ on success. FWHM $= 2\sqrt{2\ln2}\cdot\sigma \approx 2.3548\sigma$.
- **Code**: `src/engine/wasm_compute/src/photometry.rs:9-152` (`fit_gaussian_2d`), `:158-219` (`refine_stars_bulk_impl`, per-star stamp extraction); caller `src/engine/pipeline/m10_psf/psf_field.ts:39` (FWHM constant), `:12-19` (fitter contract + honest degrade to moment fallback when no compiled fitter is present).
- **Cite**: Levenberg (1944), Quart. Appl. Math. 2, 164; Marquardt (1963), J. Soc. Indust. Appl. Math. 11(2), 431. Profile confirmed **Gaussian** — no Moffat β-profile anywhere in this file/module (grep-confirmed); Moffat (1969) does NOT apply here.
- **Gotcha**: no gain-ratio / chi-square-decrease test gates step acceptance — `photometry.rs:137` comment states "Skipping rigorous chi-square check for raw performance"; every step with an invertible $(J^TJ+\lambda I)$ is accepted unconditionally, only λ's direction depends on invertibility. This is a deviation from textbook LM's trust-region acceptance rule. F also carries NO pedestal/background term — the caller must background-subtract each stamp first (`psf_field.ts:14,371`), else the fit is biased.

### 3. Richardson-Lucy deconvolution (damped, nebulosity-protected)
An optional, default-off diagnostic sharpening step: iteratively sharpens a bounded local window around a star, with several house additions (damping, a ratio clamp, a saturated-pixel freeze, and a diffuse/compact split) layered on top of the textbook algorithm to keep it from producing artifacts. It never alters the displayed image by default and never touches the coordinate ledger.
- **Base update**: est ← est · corr(obs / conv(est,K), K) (standard multiplicative RL). This code additionally: dampens the ratio smoothstep-toward-1 within 1.5·σ_noise of unity, clamps ratio∈[0.25,4], freezes saturated pixels to the observed value every iteration, and (protected variant) splits the window into a box-blurred DIFFUSE component + a COMPACT residual so RL only sharpens the compact part.
- **Code**: `src/engine/pipeline/m10_psf/rl_deconv.ts:180-230` (`richardsonLucyWindow`, damping `:200-211`), `:154-177` (`richardsonLucyWindowProtected`, diffuse/compact split), `:105-134` (`boxBlur`).
- **Cite**: Richardson (1972), "Bayesian-Based Iterative Method of Image Restoration", JOSA 62(1), 55; Lucy (1974), "An iterative technique for the rectification of observed distributions", AJ 79, 745.
- **Gotcha**: NOT vanilla RL — damping band, ratio clamp, saturated-core freeze, and diffuse/compact splitting are all local additions, explicitly labeled APPROXIMATE (`PsfReport.approximate`). Windowed (bounded local stamps, not full-frame) with clamp-to-edge convolution; **default OFF**; runs on the NATIVE-grid PIXEL ledger only, never touches WCS, and never alters the image the user is shown unless explicitly enabled and inspected as a diagnostic. The nebulosity-**protected** variant (`richardsonLucyWindowProtected`) is the wired path behind that same default-OFF flag — still gated, still APPROXIMATE, still subordinate to the render-ML labeling/consent rules (gated / labeled-as-approximate / default-off). Known artifact stands: RL rings on ultra-deep stacks (annular ringing — see this card's annular note below).

### 4. Forced photometry at catalog positions + scrambled-null escalation
After a solve, measures flux at exactly the pixel position the catalog predicts for each candidate star (rather than re-detecting it), and separately measures the same statistic at random scrambled positions to build an empirical "how often would this happen by chance" null.
- **SNR**: $snr = flux / noise$; $flux = apSum - n_{ap}\cdot bg_{median}$; $noise = \sigma_{local}\cdot\sqrt{n_{ap} + n_{ap}^2/n_{bg}}$ — aperture Poisson-like term PLUS the propagated uncertainty of the annulus background mean itself. Background: median + $MAD\cdot1.4826$ sigma-clip (one 3σ clip round) over annulus $[r_{ap}+3, r_{ap}+8]$, widened until ≥40 px.
- **Escalation statistic**: binomial-excess $z = (observed - n\cdot p_0) / \sqrt{n\cdot p_0\cdot(1-p_0)}$, $p_0 = \max(nullRate, 0.5/nullN)$; null calibrated from `scrambles`(=5) draws of deterministic (mulberry32 PRNG) uniform on-frame positions through the SAME aperture/threshold as the real measurement.
- **Code**: `src/engine/pipeline/m6_plate_solve/deep_verify.ts:91-136` (`forcedMeasure`, noise formula `:134-135`), `:120-130` (sigma-clip), `:207-214` (`sampledBackgroundSigma`), `:271-` (mulberry32), `:335-341` (`binomialExcessZ`), `:368-428` (`deepVerifyEscalation`).
- **Cite**: aperture-noise form is a restricted case of the standard CCD aperture-photometry noise equation (cf. Howell, *Handbook of CCD Astronomy*, 2nd ed., ch. 4 "CCD equation") — background + background-uncertainty terms only, no explicit source-Poisson/dark-current/read-noise terms (folded into the measured $\sigma_{local}$). The binomial-excess z is a standard one-proportion z-test (normal approximation to the binomial), applied here as a bespoke on-frame empirical-null calibration rather than a named detection statistic.
- **Gotcha**: MAD→σ constant 1.4826 assumes a Gaussian background (standard practice, unstated in-code). The scrambled null MUST reuse the identical aperture/threshold as the real measurement for the "chance rate" to be apples-to-apples — changing one side without the other silently breaks the statistic.

### 5. Set-level confirmation gate ("family-wise", actually bespoke)
Decides, for the whole candidate set at once, whether enough forced-photometry measurements came back significant to call the solve confirmed — despite the in-code naming, this is a single aggregate statistical test, not a classical multiple-comparisons correction.
- **Formula**: $setExcessZ = binomialExcessZ(realConfirmed, N, nullConfirmRate, nullTotal)$ — the SAME one-proportion z-test as #4, run ONCE over the whole candidate set vs. a frame-wide scrambled-null confirm rate (measured from `setNullDraws`(=4) draws through the IDENTICAL per-star confirm predicate: SNR pass + neighbor pass + shape/local-null not-FAIL). Pass ⟺ $z \ge setExcessGateZ$ AND $realConfirmed > expected$.
- **Code**: `src/engine/pipeline/m6_plate_solve/forced_confirm.ts:362-428` (`confirmForcedSet`), shared statistic at `deep_verify.ts:335-341`; live threshold wired at `solver_entry.ts:2528` (`setExcessGateZ: PC.SOLVER_CONFIRM_SET_EXCESS_Z`).
- **Cite**: none — this is BESPOKE. Despite the name "family-wise gate" (and in-code comments invoking "family-wise excess"), there is no Bonferroni / Holm / Šidák / Benjamini-Hochberg (FDR) correction anywhere in the file — it is a single aggregate binomial-proportion z-test comparing the real set's confirm rate to an empirically-measured on-frame null rate, not a classical multiple-comparisons family-wise-error-rate procedure. Naming it "family-wise" is a code-comment idiom, not a statistical claim to take literally.
- **Gotcha**: two default values coexist — the module's own `DEFAULT_CONFIRM_CONFIG.setExcessGateZ=10` (`forced_confirm.ts:129`) vs. the ONLY live caller's override to `PC.SOLVER_CONFIRM_SET_EXCESS_Z=15` (calibrated — see docs/GATES.md; **N=1-calibrated, SeeStar-only**, not yet a validated science-product threshold). Reading this file in isolation understates the real live gate by Δz=5. **Note (superseded elsewhere):** this z-based gate is itself frozen/reported-only in the current confirm lane, which decides set-level confirmation via a Benjamini-Yekutieli FDR step-up procedure instead (see `forced_confirm.ts`) — this card documents the mechanism named above as it exists in this file; the FDR gate is a separate, newer mechanism this card does not yet cover.

**"Annular" — two distinct meanings across the cards (disambiguation for fan-out):** (1) **annulus background** = the sky ring for aperture photometry: median (+ MAD·1.4826 one-round 3σ clip) over an annulus outside the aperture — #4 here (`deep_verify.ts`, ring `[r_ap+3, r_ap+8]` widened until ≥40 px) and CARD_COLOR_PHOTOCAL #3 (SPCC RGB photometry, per-channel MEDIAN sky ring `r2..r3`). Robust background estimator, NOT an artifact. (2) **annular ring ARTIFACT** = the ringing RL deconvolution introduces on ultra-deep stacks — #3 here, a KNOWN failure mode the damping/protected variant mitigates, default-OFF. When a fan-out says "annular", pin which one: the cards default to meaning (1) unless the context is RL/deconvolution.
