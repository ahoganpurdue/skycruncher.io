# Color & Photometric Calibration — algorithm card (SPCC, gain, color-space lane)

This card covers everything that turns raw sensor counts into calibrated color and magnitude: the gain ladder from ISO/vendor register to electrons, aperture photometry, the SPCC regression that ties instrumental color to Gaia catalog color, the render-lane white-balance/color-matrix machinery, and the color-space math (Planckian locus, sRGB, Oklab) underneath it all. The atlas's only color axis is Gaia BP−RP; Johnson B−V is not measured or stored anywhere in this pipeline, and shows up in code only as a legacy field name or as an external literature convention some formulas were originally derived for.

Symbols: ADU=raw counts, e−=electrons, gain=`gain_e_adu` (e−/ADU, delivered bit depth), norm=ADU-range-normalized flux ∈[0,1], m=magnitude, BP−RP=Gaia DR3 color index (mag), B−V=Johnson color index (mag, external-literature convention only — see entry 6), T=temperature (Kelvin), (x,y)=CIE 1931 chromaticity, n_pix=aperture pixel count, MAD=median absolute deviation.
Every entry below is VERIFIED against the code at the cited line; formulas transcribed from the running code, not idealized from the literature.

### 1. SPCC color regression (instrumental color → catalog BP−RP)
Fits a straight line from a star's measured instrumental color (a ratio of its blue and red aperture fluxes) to its known Gaia BP−RP catalog color — this fit is the whole basis for the render-lane color calibration described later in this card.
- **Formula**: model $catBpRp = slope\cdot instColor + intercept$, with $instColor = -2.5\log_{10}(flux_b/flux_r)$ (built at `spcc_calibrator.ts:229`). Closed-form OLS per round: $slope = (n\Sigma xy - \Sigma x\Sigma y)/(n\Sigma xx - (\Sigma x)^2)$, $intercept = (\Sigma y - slope\cdot\Sigma x)/n$; degenerate guard $|denom| < 10^{-12}$ → invalid. Residual clip: $\sigma = \sqrt{\Sigma r^2/n}$ (RMS about the fit, denominator **n**, no n−2 dof correction), keep $|r| \le sigmaClip(=2.5)\cdot\sigma$, up to $maxIter(=3)$ fit+clip rounds; early exit when $\sigma \le 10^{-12}$ or nothing is clipped. `minStars`=8 enforced before every fit and again after the loop. Final stats on the surviving set: $R^2 = 1 - SS_{res}/SS_{tot}$ ($SS_{tot} \le 10^{-12} \to R^2=0$), $RMSE = \sqrt{SS_{res}/n}$. Invalid fit ships identity defaults slope=1, intercept=0.
- **Code**: `src/engine/pipeline/m8_photometry/spcc_calibrator.ts:100-154` (`fitColorRegression`; defaults `:104-106`, OLS `:115-127`, clip `:129-137`, final stats `:141-153`); invalid sentinel `:86`. Live caller: `src/engine/pipeline/stages/science.ts:58`.
- **Cite**: catalog truth is the Gaia (E)DR3 BP/RP photometric system — Riello et al. (2021), "Gaia Early Data Release 3: Photometric content and validation", A&A 649, A3. The OLS+k·σ-clip machinery is textbook (cf. Press et al., *Numerical Recipes*, 3rd ed., §15.2 — book, not peer-reviewed). "SPCC" naming lineage: PixInsight's SpectroPhotometric Color Calibration [VERIFY — commercial software documentation, not peer-reviewed; this is an independent, much simpler implementation — no filter-curve × spectrum synthesis, just a linear fit of instrumental color to catalog color].
- **Gotcha**: OLS puts all error on Y, but here Y (catalog BP−RP) is near-noiseless and X (instrumental color) is the noisy one — inverted from the errors-in-variables ideal, so faint-star noise attenuates the slope (regression dilution); no total-least-squares treatment. Clip σ is a plain RMS, not the MAD-robust σ its zero-point sibling (#2) uses. `matched.catalog.bv` **IS** Gaia BP−RP, not Johnson B−V, despite the field name (`spcc_calibrator.ts:11-13`; see UNIT WARNINGS).

### 2. SPCC zero-point (clipped median with MAD-robust σ)
Anchors the absolute magnitude scale for a frame: a robust, outlier-resistant offset between instrumental magnitude and Gaia G, computed once per frame and absorbing both gain-scale error and single-frame atmospheric extinction.
- **Formula**: $d_i = catG_i - mInst_i$; $zp = \operatorname{median}(d)$; $MAD = \operatorname{median}(|d - zp|)$; $\sigma = 1.4826\cdot MAD$. ONE clip round: keep $|d - zp| \le sigmaClip(=2.5)\cdot\sigma$, applied only when $\sigma > 10^{-12}$ AND survivors ≥ `minStars`(=5) AND at least one sample was actually removed; zp/MAD/σ then re-derived on the survivors. Returns {zeropoint: zp, rmse: σ, n_used}.
- **Code**: `src/engine/pipeline/m8_photometry/spcc_calibrator.ts:160-185` (`fitZeroPoint`; defaults `:164-165`, median/MAD `:170-172`, single clip `:174-182`); `median` helper `:89-94` (even-n averages the middle pair).
- **Cite**: 1.4826 = $1/\Phi^{-1}(3/4)$, the Gaussian consistency constant for MAD→σ — Rousseeuw & Croux (1993), "Alternatives to the Median Absolute Deviation", JASA 88(424), 1273; Hampel (1974), JASA 69, 383. (Same constant, same assumption, as `deep_verify.ts` — see CARD_PHOTOMETRY_STATS.md #4.)
- **Gotcha**: the output field is named `rmse` but carries the **robust σ (1.4826·MAD)**, not a root-mean-square error — the docstring (`:156-158`) says so, the field name doesn't. By design the zero point absorbs the gain LUT's absolute-scale error AND single-frame atmospheric extinction (`:15-17`); it is a per-frame fit constant, NOT a sensor property — `air_mass` is recorded upstream for a future multi-frame extinction solve that does not exist yet.

### 3. RGB aperture photometry feeding SPCC
Measures each star's raw red/green/blue flux with a simple aperture-and-sky-ring photometry routine — the direct input to both the SPCC color regression above and the zero-point fit.
- **Formula**: aperture radius $r_1 = \max(3.5, 1.5\cdot FWHM_{px})$ (FWHM fallback 3.0 px), dead zone to $r_2 = r_1+1.5$, sky ring $r_2..r_3 = r_2+3.0$. Hard-edged membership on pixel centers: $dist^2 \le r_1^2$ (no fractional-pixel weighting). Per channel $c \in \{r,g,b\}$: $flux_c = \sum_{aperture} c - \operatorname{median}(sky_c)\cdot n_{aperture}$ — background is the per-channel **median** of the sky ring (robust to star wings/hot pixels, `:9-12`). Saturation: any channel > `SATURATION_NORM` = 0.97 (normalized) anywhere in the aperture.
- **Orchestration** (`spcc_calibrator.ts:198-261`, `computeSpccCalibration`): detected preview coords → native via `scales.previewToNative(x,y)`; `scales === null` ⇒ identity 1:1 (`:211-213`). Center-bounds exclusion native.x/y ∈ [1, dim−1) (`:216`). Usability gates: $n_{aperture} > 0\ \&\&\ n_{annulus} \ge 8\ \&\&\ !saturated$ (`:227`); color sample additionally needs $flux_b > 0\ \&\&\ flux_r > 0$ (`:228`), zero-point sample needs $flux_g > 0$ (`:231`). Per-star results index-aligned with the input matched-star list; $valid = colorFit.valid \lor zpFit.valid$ (`:255`).
- **Code**: `src/engine/pipeline/m8_photometry/rgb_aperture_photometry.ts:60-138` (`measureApertureRGB`; radii `:71-73`, membership `:105-118`, background subtraction `:122-129`), constant `:38`.
- **Cite**: concentric aperture/annulus sky photometry lineage — Stetson (1987), "DAOPHOT: A computer program for crowded-field stellar photometry", PASP 99, 191 — lineage only; independent implementation (median background, not DAOPHOT's mode estimator; no PSF fitting here).
- **Gotcha**: only the aperture **center** is bounds-checked (1-px margin); for stars near the frame edge the aperture and annulus loops silently clamp to the frame (`:79-82`) — flux is truncated/underestimated rather than the star being excluded. `AnnulusMetrology` is deliberately NOT reused (`:13-16` — single-channel, electron-domain, profile-coupled). All three fluxes stay in normalized 0..1·pixel units; only the G channel is converted to electrons downstream (#4).

### 4. Instrumental magnitude, CCD SNR equation, ISO→gain ladder
Converts raw ADU counts to electrons and then to an instrumental magnitude, and separately implements the standard CCD signal-to-noise equation and the ISO-to-gain lookup for DSLR frames (FITS frames get gain from vendor metadata instead — entry 5).
- **LIVE FILE**: `src/engine/pipeline/m8_photometry/photometry_manager.ts` — this is what `spcc_calibrator.ts:20`, `orchestrator.ts:47`, `metadata_reaper.ts:4`, and the m3/m4 stages import. The near-identical `src/engine/core/PhotometryManager.ts` is a **dead twin**: its only importer is `src/engine/tests/photometry.test.ts` (grep-verified).
- **Instrumental magnitude** (`photometry_manager.ts:117-134`): $signal_{ADU} = signal_{norm}\cdot(white\_level - black\_level)$; $signal_e = signal_{ADU} \times gain\_e\_adu$; $m = -2.5\log_{10}(signal_e/(exposureTime\lor1))$; $signal_e \le 0$ → sentinel 20. The `aperture` parameter is accepted and unused (`:131`).
- **CCD SNR** (`:79-95`): $signal_e = signal_{ADU}\cdot gain$; $bg_e = bg_{ADU}\cdot gain\cdot n_{pix}$; $noise = \sqrt{signal_e + bg_e + rn^2\cdot n_{pix}}$, i.e. $SNR = S/\sqrt{S + n_{pix}\cdot(B + \sigma_{read}^2)}$ with S=source e−, B=per-pixel sky e−, $\sigma_{read}$=`read_noise_e`. No dark-current term, no source $n_{pix}$ factor (S is the integrated flux).
- **ISO ladder** (`:41-45`): $gain = 8.0/(iso/100) \equiv 8.0\cdot(100/iso)$ e−/ADU; iso ≤ 0 (missing EXIF) → 0.5 fallback. Live for CR2/NEF EXIF path via `metadata_reaper.ts:244-247`; FITS path bypasses it entirely (#5). Default profile (`:26-35`): gain 0.5, read noise 2.5 e−, black 8192, white 61440, bit_depth 16, pixel 4.3 µm.
- **Cite**: CCD SNR equation — Howell (1989), "Two-dimensional aperture photometry: signal-to-noise ratio of point-source observations and optimal data-extraction techniques", PASP 101, 616 (restricted form: sky + read noise only, no dark/digitization terms). Logarithmic magnitude scale — Pogson (1856), MNRAS 17, 12. ISO→gain ladder: [VERIFY — empirical Canon APS-C (T6-class) characterization; matches hobbyist sensor-measurement practice, no peer-reviewed source].
- **Gotcha (convention trap, the reason this card exists)**: the two twins use **opposite gain conventions**. The live m8 file computes $e- = ADU \times gain\_e\_adu$ (`:88,:127`) — dimensionally correct when gain is e−/ADU as documented. The dead core/ twin computes $e- = ADU \div gain\_e\_adu$ (`core/PhotometryManager.ts:93,:115`) with the same "e−/ADU" doc — inverted; resurrecting it would silently flip the gain ladder's effect on every absolute flux. Pinned here: **the shipping convention is multiply**. Further drift, both twins: the SNR docstrings write $SNR = S/\sqrt{S+B+\sigma^2}$ omitting the $n_{pix}$ factor the code applies; the core twin's `getGainForISO` comment (`:48-49`) describes a sub-linear power-law fit (p≈0.95) while both implementations are plain linear reciprocal; invalid-magnitude sentinel is 20 in m8 vs 99.0 in core. Absolute e− values are only as good as the gain LUT (#5) — SPCC's zero point (#2) absorbs the scale error for calibrated output.

### 5. Vendor gain-setting LUT (ZWO GAIN register → e−/ADU)
Converts a ZWO camera's dimensionless GAIN register setting (found in the FITS header) into an actual e−/ADU gain value, via a piecewise-linear interpolation over a vendor-published curve — this is how FITS-lane (SeeStar) frames get their gain, in place of the ISO ladder that serves DSLR frames.
- **Formula**: `gain_curve` = [(setting, e_adu_native)] pairs at the **native 12-bit ADC**: IMX585 (`sensor_db.ts:395-401`): (0, 6.55), (100, 2.0), (200, 0.65), (252, 0.55), (300, 0.31). Resolution (`:527-545`): piecewise-linear interpolation in `setting`, clamped to curve endpoints; delivered gain = $e\_adu_{native} / 2^{adu\_bit\_shift}$ — shift is 4 for all three Seestar profiles (12-bit ADC left-shifted into a 16-bit FITS container ⇒ ÷16); returns null when the profile has no curve.
- **Wiring**: `metadata_reaper.ts:295-310` — FITS `GAIN` card + `findSensorByCamera(camera_model)` → `getGainForSetting`; no sensor match / non-numeric `GAIN` card / no curve ⇒ fallback gain 0.05 e−/ADU (guard `:298-301`, fallback `:309`); `BIAS` card → black_level, white_level pinned 65535, pixel size from `XPIXSZ` else 2.9 µm. Profiles: IMX585 (Seestar S30 Pro / ASI585MC, `:364-403`), IMX462 (Seestar S50, `:414-444`), IMX662 (Seestar S30 non-Pro, `:450-480`). Test coverage: `src/engine/tests/m2_seestar_profile.test.ts` (setting 150 → 1.325/16; clamp −50 → 6.55/16, 900 → 0.31/16; IMX571 → null).
- **Cite**: [VERIFY — ZWO's published ASI585MC gain chart (vendor datasheet/marketing curve), not peer-reviewed; transcribed values above are the code's literals]. IMX462/IMX662 **gain curves are borrowed verbatim from IMX585** (their `qe_curve`s are red-end-adjusted copies, NOT verbatim, despite the borrow comment naming qe_curve too) — the code flags this itself: "APPROXIMATE PHOTOMETRY … expect ~20-30% gain-scale uncertainty vs a real chart" (`:410-413`).
- **Gotcha**: the FITS `GAIN` card is a ZWO **register setting** (dimensionless), never an ISO and never e−/ADU — guard comments exist at `fits_decoder.ts:317` and `sensor_db.ts:51` ("never feed the setting to getGainForISO"). Camera lookup is longest-match, not first-match: 'Seestar S30' ⊂ 'Seestar S30 Pro' are DIFFERENT sensors (IMX662 vs IMX585); `findSensorByCamera` (`:490-505`) collects all substring matches and the longest body name wins.

### 6. BP−RP ↔ B−V 0.85 proxy, Ballesteros T_eff, spectral binning, fallback color table
The atlas's only stored color is Gaia BP−RP; this entry documents every place BP−RP gets converted to a temperature or an approximate Johnson B−V, using a literature formula (Ballesteros) that was itself derived for B−V, not BP−RP — and flags the resulting stale prose still living beside the code.
- **Formula** (`color_index.ts:95-98`): $proxyBv = 0.85\cdot bpRp$; $T = 4600\cdot\left(\dfrac{1}{0.92\,proxyBv + 1.7} + \dfrac{1}{0.92\,proxyBv + 0.62}\right)$. Teff twin (m6 `stellar_classification.ts:114-118`, in `m6_plate_solve/`, not m8): identical Ballesteros form applied to **raw B−V**, no proxy factor. RGB→BP−RP estimator (`color_index.ts:69-88`): $xyz = srgbToXYZ(r,g,b)$; $blueRatio = Z/(X + 0.001)$; $BP\_RP = -1.65\ln(blueRatio + 0.01) + 0.15$ (all four constants empirical, in-house). MK binning (`:103-111`): O > 30000 K, B > 10000, A > 7500, F > 6000, G > 5200, K > 3700, else M. Catalog fallback (m6 `m6_plate_solve/star_catalog_adapter.ts:357-367`, applied `:378` when a row lacks `bp_rp`): O=−0.40, B=−0.20, A=0.05, F=0.45, G=0.82, K=1.50, M=2.30; unknown/missing type → 0.82 (≈solar).
- **Code**: `src/engine/pipeline/m8_photometry/color_index.ts:69-111`; twins/fallback as cited above. Exported library surface (`Mainlogic_entry.ts:69-76`); not on the SPCC critical path (#1 uses catalog BP−RP directly).
- **Cite**: Ballesteros, F.J. (2012), "New insights into black bodies", EPL 97, 34008 (peer-reviewed; the formula is derived for **Johnson B−V**, not Gaia BP−RP — this is an external-literature use of B−V, not a claim that the pipeline measures B−V). 0.85 proxy factor: in-house empirical constant, no external source [VERIFY]. Fallback color table: in-house rounded means "derived from G-band and B-V approximations" (`:361`) — same spirit as the Pecaut & Mamajek (2013, ApJS 208, 9) spectral-type color tabulations but NOT transcribed from them [VERIFY].
- **Gotcha**: the file's prose is B−V-era (concept header "The Color Index (B-V)" `:7`, "estimated B-V" `:67`, Vega/Betelgeuse B−V anchors `:64`) while the code returns `BP_RP` — stale documentation inside the file itself, not a code bug. `compareToStandard` (`:134`) diffs an observed BP−RP against `StandardStar.color_index_BV`, and that field actually **carries BP−RP** for Gaia atlas rows (`star_catalog_adapter.ts:392` — the field is legacy-named, flagged in-code for an eventual rename to `bp_rp_index`): values consistent, field name a trap. There is no live path anywhere in this codebase that measures or stores a real Johnson B−V from image data or from HYG — the atlas is Gaia-pure (Gaia G≤15 plus a Tycho-2/Hipparcos bright-star supplement), and any `HYG_`-prefixed code still visible in a grep is a dead defensive fallback that never fires against shipped data. Stacking proxy(0.85) → Ballesteros(B−V-derived) makes T_eff a rough estimate (few-hundred-K class), fine for MK binning, not for science temperatures.

### 7. Planckian locus xy cubics + sRGB EOTF/OETF + XYZ↔sRGB + Oklab + CIE Lab
The general-purpose color-space toolbox this pipeline shares across consumers: black-body color as a function of temperature, sRGB's gamma curve, and conversions between XYZ, sRGB, Oklab, and CIE Lab.
- **Planckian locus** (`colormath.ts:42-58`, `planckianXY`): $x_c = 1000/T$. x: $T \in [1667, 4000]$: $x = -0.2661239x_c^3 - 0.2343589x_c^2 + 0.8776956x_c + 0.179910$; **else** (covers T > 4000 AND, unguarded, T < 1667): $x = -3.0258469x_c^3 + 2.1070379x_c^2 + 0.2226347x_c + 0.240390$. y: $T \in [1667, 2222]$: $y = -1.1063814x^3 - 1.34811020x^2 + 2.18555832x - 0.20219683$; $T \in (2222, 4000]$: $y = -0.9549476x^3 - 1.37418593x^2 + 2.09137015x - 0.16748867$; else: $y = 3.0817580x^3 - 5.87338670x^2 + 3.75112997x - 0.37001483$. `planckianLocus` (`:106-122`): log-spaced sweep 1667→25000 K, default 100 steps.
- **sRGB → XYZ** (`:14-24`): channels ÷255, EOTF $v > 0.04045\ ?\ ((v+0.055)/1.055)^{2.4}\ :\ v/12.92$; matrix rows (0.4124, 0.3576, 0.1805 | 0.2126, 0.7152, 0.0722 | 0.0193, 0.1192, 0.9505). **XYZ → sRGB** (`:27-39`): matrix (3.2406, −1.5372, −0.4986 | −0.9689, 1.8758, 0.0415 | 0.0557, −0.2040, 1.0570); OETF $v > 0.0031308\ ?\ 1.055v^{1/2.4} - 0.055\ :\ 12.92v$; output clamped+rounded to 0–255 integers.
- **XYZ → Oklab** (`:71-85`): $LMS = M_1\cdot XYZ$ with $M_1$ rows (0.8189330101, 0.3618667424, −0.1288597137 | 0.0329845436, 0.9293118715, 0.0361456387 | 0.0482003018, 0.2643662691, 0.6338517070); nonlinearity $\sqrt[3]{\max(0,\cdot)}$; $Lab = M_2\cdot LMS'$ with $M_2$ rows (0.2104542553, 0.7936177850, −0.0040720468 | 1.9779984951, −2.4285922050, 0.4505937099 | 0.0259040371, 0.7827717662, −0.8086757660).
- **XYZ → CIE Lab** (`:88-103`): D65 white $X_n=0.950489, Y_n=1.000000, Z_n=1.088840$; $f(t) = t > (6/29)^3\ ?\ t^{1/3}\ :\ (1/3)(29/6)^2 t + 4/29$; $L=116f_Y-16$, $a=500(f_X-f_Y)$, $b=200(f_Y-f_Z)$.
- **Code**: `src/engine/core/colormath.ts:14-122`. Live consumers: `standard_stars.ts:71` (expected star xy from T), `zenith.ts:312`, `color_index.ts:72`, Oklab ΔE at `color_index.ts:137-154`.
- **Cite**: Planckian cubic approximations — Kim et al. (2002), "Design of Advanced Color Temperature Control System for HDTV Applications", J. Korean Phys. Soc. 41(6), 865 [VERIFY — coefficients match the widely-circulated secondary form (Wikipedia "Planckian locus"); primary paper not checked against the code]. sRGB EOTF/OETF and D65 matrices — IEC 61966-2-1:1999 (industry standard, not a paper); the 4-decimal matrix pair is the common truncation. Oklab — Ottosson (2020), "A perceptual color space for image processing" [VERIFY — self-published technical note; de-facto standard (adopted by CSS Color 4) but NOT peer-reviewed]. CIE Lab — CIE 15 *Colorimetry* / ISO/CIE 11664-4 (D65 2° white).
- **Gotcha**: `srgbToXYZ` expects **0–255** channels (÷255 at `:15`), but `computeColorIndex`'s contract says "hardware-normalized sRGB" — 0..1 inputs all land below the 0.04045 knee and take the linear /12.92 branch, silently bypassing the EOTF (ratios like Z/X partially survive, absolute values don't). `planckianXY` has NO lower-range guard: T < 1667 K falls into the high-T x-branch and extrapolates garbage. Oklab's $\sqrt[3]{\max(0,\cdot)}$ clamp is a house deviation from the reference implementation (prevents NaN for out-of-gamut negative LMS). The forward/inverse sRGB matrices are independently rounded — round-trips are approximate, further quantized by the 0–255 clamp+round in `xyzToSRGB`. `compareToStandard` (`color_index.ts:144-148`) re-derives XYZ from xy inline instead of calling `xyToXYZ` (`:61-68`) — duplicate math, same formula.

### 8. SPCC channel-gains TLS/EIV estimator + render-lane white-balance application
Fits per-channel multiplicative white-balance gains so that a frame's instrumental color matches Gaia truth, using total-least-squares (not ordinary least squares, which would bias the fit) — these gains are the render-lane's preferred white-balance source when they pass a quality gate.
- **Formula**: per-channel multiplicative render WB gains fitted so instrumental color matches Gaia truth, estimated by **TLS / errors-in-variables** (principal-axis fit), never OLS — the header (`spcc_calibrator.ts:296-304`) records why: OLS attenuates the color-axis slope **~33% toward zero** (measured TLS slope 1.326 vs. attenuated OLS), which would bias the gains. Gains are green-anchored (WHITE_REF_BP_RP=0.0 ≈ A0V/Vega). Interface `SpccChannelGains` (`:315-327`): `gains:[r,g,b]`, `method:'TLS'` always, binding quality $r^2=\min(r^2_{b-r}, r^2_{g-r})$. Quality gate constants (`pipeline_config.ts:451-462`): `SPCC_GAINS_MIN_STARS=20`, `SPCC_GAINS_MIN_R2=0.55`, `SPCC_GAINS_SLOPE_MIN=0.3`/`MAX=3.0` (b−r TLS slope sanity), `SPCC_GAINS_MIN_GAIN=0.25`/`MAX=4.0` (per-channel sanity, MAX catches a measured gain_b=145.7 pathology on one test frame), `SPCC_GAINS_APPLY` default **ON** with env kill-flag (`SPCC_GAINS_APPLY=0`). g−r slope held to the SAME bound as b−r.
- **Application (RENDER-LANE ONLY, PIXEL ledger)**: `ImageProcessor.ts:133-202` — when `spccWb.applied`, the catalog TLS gains **REPLACE** the star-ensemble-white heuristic (#7's two-part cal) inside the same `[0.25,8]` safety clamp (`:199-200`); with no applied gains the render is **byte-identical** to the pre-gains path (`:142-143`). Composition order (`:133-141`): a body matrix (#9) rotates cam-RGB→linear-sRGB FIRST, then SPCC gains are the residual WB in that space. They never co-occur today (SPCC is FITS-only ⇒ SeeStar/IMX462 which has NO matrix; matrix bodies are CR2/DSLR where SPCC does not run — a documented forward design decision).
- **Code**: estimator + interface `src/engine/pipeline/m8_photometry/spcc_calibrator.ts:296-327`; constants `constants/pipeline_config.ts:451-462`; render application `src/engine/core/ImageProcessor.ts:133-202`. Always-record `spcc.gains` receipt block; the overall `RECEIPT_SCHEMA_VERSION` has moved on since — cite the const in `schema_versions.ts`, never this card, for the live value.
- **Cite**: total least squares / errors-in-variables (Deming-style orthogonal regression) — standard EIV treatment (cf. Fuller 1987, *Measurement Error Models*, Wiley); TLS closed form is the smaller-eigenvalue principal axis of the centered 2×2 moment matrix (`spcc_calibrator.ts:270`). Gaia truth as #1.
- **Gotcha**: the calibration sample this gate was validated against was small and bimodal — only two of seven candidate frames passed the R² gate and were applied (M66 R²=0.897, M81 R²=0.757), five were honestly refused; `MIN_R2=0.55` sits in the empirical gap `[0.373, 0.757]` between the refused and accepted frames' R² values — it is a small-sample fit, NOT a validated science threshold. This is additive, always-record evidence and stays gate-safe: gains touch RENDER pixels only, the solve/receipt WCS never.

### 9. Camera-RGB → linear-sRGB forward color matrix (`camera_color_matrix.ts`)
A per-camera-body color matrix (sourced from the same reference tables raw-photo tools use) that rotates a DSLR's native color response into linear sRGB before the display stretch — applied only to cameras with a known matrix; everything else honestly falls back to the empirical white-balance path instead of guessing.
- **Formula**: per-body `ColorMatrix2` (XYZ D65 → camera-raw, ×10⁴, 3×3 row-major) from the widely-used dcraw `adobe_coeff[]` table (mirrored by LibRaw's `colordata.cpp`). `buildTransforms` (`:151`) inverts to get camera-raw→XYZ then composes with `XYZ_RGB` (`:31-35`, the standard linear-sRGB→XYZ D65 matrix) to yield the cam-RGB→linear-sRGB rotation applied before the STF stretch. `resolveMatrix(make,model)` (`:104`) does alias-normalized lookup; `resolveColorTransform(model)` (`:205`) returns the composed transform or **null**.
- **Bodies**: `CAMERA_MATRICES` (`:48`) 5D3 `[6722,-635,-963,-4287,12460,2028,-908,2162,5668]`, 60D, **60Da distinct** (`:59`, NOT a 60D fallback — the Hα-modified body, whose large-red primary lands honestly **out-of-gamut**). `NO_MATRIX_BODIES` (`:80`): SeeStar / IMX462 and every non-DNG astro-cam ⇒ `resolveColorTransform`→null ⇒ render falls back to #7's empirical luminance/white-balance path (honest-or-absent — never a fabricated matrix). ### 10. SPCC color-fidelity report (survivor-vs-unclipped + TLS/OLS attenuation bracket) — research, never a gate
A self-reporting quality check on the SPCC fit itself: it reports both the flattering (sigma-clipped) fit quality and the honest full-set number side by side, so the attenuation bias entry 1 describes can actually be quantified. Research-only — it never gates anything.
- **Formula** (`computeColorFidelity`, `spcc_calibrator.ts:242-290`): reports BOTH the σ-clipped **survivor-only** $r^2$/RMSE (optimistic — the engine's fit view) AND the **full-set unclipped** $r^2$/RMSE (the honest counterpart), PLUS a TLS/EIV slope bracket: $slope_{tls} = \dfrac{S_{yy} - S_{xx} + \sqrt{(S_{yy}-S_{xx})^2 + 4S_{xy}^2}}{2S_{xy}}$ (`:270`) with $attenuation\_ratio = slope_{tls} / ols_{yx}$ (`:272`; >1 ⇒ OLS under-estimates |slope|).
- **Code**: engine `spcc_calibrator.ts:242-290` (interface `ColorFidelity` `:204-241`); thin CLI driver `tools/color/star_color_fidelity.mjs` (survivor optimistic vs. `r2_unclipped`/`rmse_mag_unclipped`, `slope_bracket_eiv`). Additive `spcc_fidelity` receipt block.
- **Cite**: regression-dilution / attenuation bias under errors-in-variables — same EIV lineage as #8; the survivor-vs-unclipped honesty split is in-house.
- **Gotcha**: a research block, N=1, `validated:null`, NEVER a gate — it is the honest-reporting sibling to #1's OLS fit (which has the attenuation problem this block quantifies but does NOT correct in the fit). Survivor $r^2$ (e.g. ~0.96 on a good frame) is systematically optimistic vs. the unclipped full-set $r^2$; quote the unclipped number for science.

## UNIT WARNINGS
- **`magnitude_V` carries Gaia G for Gaia-format rows** (`star_catalog_adapter.ts:494`: `magnitude_V = s.mag_g ?? s.mag`) — a per-row `band` tag (`:498`, `'GaiaG'` vs `'JohnsonV'`) discriminates, but the FIELD NAME lies. **OPEN BUG:** the Gaia-G-in-`magnitude_V` slot causes a **±0.2–0.5 mag silent filter/selection shift** where consumers assume Johnson V; a fix is proposed (three options under consideration: a G/V-convention fix, dedup-only, or Gaia-primary-with-supplements) but not yet decided. NOT YET FIXED.
- `gain_e_adu` is e−/ADU **in the delivered 16-bit container** (already ÷$2^{adu\_bit\_shift}$), never native 12-bit-ADC units (`sensor_db.ts:521-545`).
- Electron conversion on the live path is $e- = ADU \times$ `gain_e_adu` (`m8_photometry/photometry_manager.ts:88,127`); the dead `core/PhotometryManager.ts` twin **divides** — do not resurrect it without fixing the convention (entry #4).
- The FITS `GAIN` header card is a dimensionless ZWO register **setting** — not ISO, not e−/ADU; route only through `getGainForSetting` (`fits_decoder.ts:317`, `sensor_db.ts:51`).
- SPCC aperture fluxes are ADU-range-**normalized** (0..1·pixel units); only the G channel is converted to electrons (inside `calculateInstrumentalMagnitude`), R/B never are — `instColor` is a pure ratio so this cancels.
- `matched.catalog.bv` **IS Gaia BP−RP** and `matched.catalog.mag` **IS Gaia G** (`spcc_calibrator.ts:11-13`); likewise `StandardStar.color_index_BV` carries BP−RP for Gaia atlas rows (`star_catalog_adapter.ts:392`) — the `bv` names are legacy field names, not a live B−V measurement.
- `srgbToXYZ` wants 0–255 channel values; `xyzToSRGB` returns clamped rounded 0–255 integers. Feeding 0..1 "normalized" pixels degenerates the EOTF to its linear branch (entry #7).
- Temperatures are Kelvin throughout; instColor, BP−RP, B−V, zp, mInst are magnitudes ($-2.5\log_{10}$ convention, factor literal at `spcc_calibrator.ts:229`).
