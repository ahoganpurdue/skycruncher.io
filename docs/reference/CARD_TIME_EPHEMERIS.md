# Time & Ephemeris — algorithm card (clock, sidereal, Kepler lane)

This card documents every place the codebase converts between calendar time, Julian Date, sidereal time, and planetary position: the clock and orbital-mechanics machinery behind timestamps, star-catalog epoch handling, and the solar-system overlay. It exists because the same physical quantities (sidereal time, precession, Kepler's equation) are implemented independently in more than one module and more than one unit convention, and those independent implementations need one place where they are checked against each other and against the running code.

Symbols: JD=Julian Date; d=days since J2000 (JD−2451545.0); T=Julian centuries ((JD−2451545)/36525); (α,δ)=RA/Dec, **RA in HOURS** unless stated; GMST/LST unit (hours vs degrees) stated per entry — this lane runs BOTH; M/E=mean/eccentric anomaly, **degrees** (Schlyter-style, not radians); ε=obliquity; π=horizontal parallax.
Every entry below is VERIFIED against the code at the cited line; none are assumed from the paper alone.

### 1. Unix→Julian Date conversion + Julian centuries/years since J2000
Converts a JavaScript timestamp into the Julian Date and the Julian-century/year values every other formula on this card builds on.
- **Formula**: $JD = \text{Date.getTime()}/86400000 + 2440587.5$ (Unix epoch 1970-01-01T00:00Z = JD 2440587.5). $T = (JD - 2451545.0)/36525$; $\text{years} = (JD - J_{2000})/365.25$; $d = JD - J_{2000}$.
- **Code**: `src/engine/core/TimeService.ts:30-33` (`toJulianDate`), `:38-40` (`toJuliancenturies`), `:45-47` (`toJulianYears`), `:52-55` (`toDayssinceJ2000`); J2000=2451545.0 at `:18`. Inverse (JD→Date) lives separately in `src/engine/pipeline/m1_ingestion/fits_decoder.ts:357` (`(jd − 2440587.5)·86400000`).
- **Cite**: standard identity; Meeus, *Astronomical Algorithms*, ch. 7 form [VERIFY — reference book, not peer-reviewed; the Unix-ms shortcut is a programming identity, not from the literature].
- **Gotcha**: the whole lane runs on **UTC treated as UT1≈TT** — no ΔT, no leap seconds, no TDB anywhere (fine at the arcminute ephemeris grade shipped). String inputs: `new Date(string)` parses a bare ISO date-time as **LOCAL** time in JS — the FITS decoder force-appends `'Z'` upstream (`fits_decoder.ts:342-346`) precisely so this method never sees a naked string; EXIF path normalizes `YYYY:MM:DD` → `YYYY-MM-DD` first (`metadata_reaper.ts:204-212`).

### 2. GMST — IAU-1982 lineage, THREE live numeric forms
Sidereal time (GMST/LST) says which point on the celestial equator is on the local meridian right now; every altitude/azimuth and hour-angle computation on this card depends on it. Three independently coded versions of the formula exist here, in two different units, that agree only to within a fraction of an arcsecond of each other.
- **Form A (degrees, continuous, full)**: $GMST^\circ = 280.46061837 + 360.98564736629\cdot(JD-2451545.0) + 0.000387933\cdot T^2 - T^3/38710000$, normalized to $[0,360)$ — `src/engine/core/TimeService.ts:62-68` (`getGMST_Deg`); $LMST^\circ = (GMST + (lon \bmod 360) + 360) \bmod 360$ at `:70-72`.
- **Form B (degrees, continuous, TRUNCATED — no $T^2$ terms)**: $GMST^\circ = (280.46061837 + 360.98564736629\cdot D) \bmod 360$ — `src/engine/pipeline/m10_psf/psf_physics.ts:148-151` (`targetGeometry`, feeds the M10 airmass/hour-angle report) and `tools/adaptive/image_conditions.ts:71`.
- **Form C (hours, linear)**: $GMST_h = (18.697374558 + 24.06570982441908\cdot d) \bmod 24$ — `src/engine/wasm_compute/src/ephemeris.rs:226` (alt/az) and `:267` (topocentric), `src/engine/wasm_compute/src/bright_star_atlas.rs:29` (`get_lst`), `src/engine/pipeline/m6_plate_solve/planetary_adapter.ts:57`; code-generated again by `tools/format_atlas_rust.ts:27`.
- **Consistency**: A, B, C share one lineage — $18.697374558\,h \times 15 = 280.46061837°$ exactly, and $24.06570982441908\,h/\text{day} \times 15 = 360.98564736629°/\text{day}$ to 12 significant figures. B and C ≡ A minus the $T^2$ terms ($\approx 0.1''$ at $T\approx0.26$ — negligible for these consumers).
- **Cite**: Aoki, Guinot, Kaplan, Kinoshita, McCarthy & Seidelmann (1982), "The new definition of universal time", A&A 105, 359 (IAU-1982 GMST). The hours-linear constants are the USNO "approximate sidereal time" reduction [VERIFY — almanac/tutorial derivation, not a separate peer-reviewed source].
- **Gotcha**: no equation of the equinoxes anywhere — GMST is used where GAST is formally required (≈±16″ ≈ 1 s of time, ignored). `planetary_adapter.ts:57` does NOT reduce gst before use (`:58` reduces lst % 24 and skips the +24 clamp) — for pre-J2000 dates lst goes negative; harmless there because the offset is exactly −24 h (−360°) and the only consumers are 360°-periodic trig (`:61-63`), but copying that LST into any NON-periodic use (clock display, differencing against another clock, a time inversion like entry 8) would break. The house trap is the UNIT, not the epoch: Form A/B return DEGREES, Form C returns HOURS.

### 3. Precession J2000↔date (Lieske ζ/z/θ) + proper motion (pmra·cosδ) + Alt/Az ⇄ RA/Dec
Precesses equatorial coordinates from a reference epoch to the date of observation, applies proper motion, and converts between local Alt/Az and equatorial RA/Dec — the coordinate-frame conversions every observing-geometry consumer on this card needs.
- **Precession (J2000→date)**: $\zeta = (2306.2181T + 0.30188T^2 + 0.017998T^3)/3600°$, $z = (2306.2181T + 1.09468T^2 + 0.018203T^3)/3600°$, $\theta = (2004.3109T - 0.42665T^2 - 0.041833T^3)/3600°$; $A = \cos\delta\sin(\alpha+\zeta)$, $B = \cos\theta\cos\delta\cos(\alpha+\zeta) - \sin\theta\sin\delta$, $C = \sin\theta\cos\delta\cos(\alpha+\zeta) + \cos\theta\sin\delta$; $\alpha' = \operatorname{atan2}(A,B) + z$ (returned in HOURS, wrapped $+24 \bmod 24$), $\delta' = \arcsin(\operatorname{clamp}(C))$. `src/engine/core/TimeService.ts:80-110`.
- **Proper motion**: pmra is Gaia-convention $\mu_{\alpha^*} = \mu_\alpha\cos\delta$ in mas/yr; $\Delta\alpha° = (pm_{ra}/3.6\times10^6)\cdot\text{years} / \cos\delta$ (guard $|\cos\delta|>10^{-6}$ else 0), applied in HOURS ($/15$); $\Delta\delta° = (pm_{dec}/3.6\times10^6)\cdot\text{years}$, dec clamped $\pm90°$. `:130-148`. Live path: `m5_coordinate_flatten/zenith.ts:87-89` applies PM first, THEN precesses.
- **Alt/Az**: $HA = LMST - \alpha$; $\sin(alt) = \sin\phi\sin\delta + \cos\phi\cos\delta\cos H$; $az = \arccos\!\left(\dfrac{\sin\delta - \sin\phi\sin(alt)}{\cos\phi\cos(alt)}\right)$, mirrored $az \to 2\pi-az$ when $\sin(HA)>0$ ⇒ azimuth from NORTH through EAST. `:155-180` (+ LMST-injected variant `:182-197`). Inverse (`horizontalToEquatorial`, `:202-231`): $\delta = \arcsin(\sin\phi\sin h + \cos\phi\cos h\cos A)$; $\alpha = LST \mp HA$ selected by sign of $\sin A$; RA returned in HOURS.
- **Cite**: Lieske, Lederle, Fricke & Morando (1977), A&A 58, 1 + Lieske (1979), A&A 73, 282 (IAU-1976 precession angles — coefficients match 2306.2181/2004.3109 exactly). PM convention: Lindegren et al. (2021), A&A 649, A2 (Gaia EDR3 astrometry, $\mu_{\alpha^*} \equiv \mu_\alpha\cos\delta$).
- **Gotcha**: the docstring at `:77` claims "(IAU 2006)" — FALSE; the live coefficients are IAU-1976/Lieske (IAU-2006 would be 2306.083227″… per Capitaine et al. 2003, A&A 412, 567). Stale comment, not a math bug (≲0.1″/century difference). `precessDateToJ2000` (`:115-123`) is NOT a true inverse — it mirrors the JD about J2000 ($jd_{inv} = J_{2000} + (J_{2000}-jd)$, i.e. $T\to -T$ in the FORWARD polynomials) instead of transposing the rotation; second-order error only, and it currently has NO production caller (grep-verified; only `precessJ2000ToDate` is wrapped by `zenith.ts:70-75`). Azimuth acos form divides by $\cos(alt)$ — singular at the zenith. No refraction anywhere in this chain (deferred per project policy). **Open question not covered by this card's verification pass**: `zenith.ts`'s own comments describe its proper-motion/precession chain as operating on "J2000 coordinates," but this codebase's live star atlas is Gaia-format at reference epoch J2016.0, not J2000 — whether that ~16-year gap is accounted for elsewhere, and whether this particular chain (`starPositionAtEpoch`) is actually reached by any live caller, was not resolved this pass (see this file's accompanying flags).

### 4. Kepler solve (5 fixed Newton steps) + Schlyter elements + magBase law + hierarchical assembly + fixed obliquity
Solves Kepler's equation for every solar-system body in the ephemeris overlay (Sun, planets, and moons) and assembles their positions into a single scene, given a fixed low-precision element set and a fixed number of solver iterations.
- **Kepler (all in DEGREES)**: $n = 0.9856076686/a^{1.5}\ °/\text{day}$ (`ephemeris.rs:82`); $M = (m + n\cdot d) \bmod 360$ (`:83-84`); seed $E_0 = M + e\cdot(180/\pi)\cdot\sin M\cdot(1 + e\cos M)$ (`:86`); then EXACTLY 5 Newton steps $E \leftarrow E - \dfrac{E - e\cdot(180/\pi)\cdot\sin E - M}{1 - e\cos E}$ (`:88-91`) — **no convergence check, no high-e guard** (max e in the DB is Pluto 0.24880766, so 5 steps converge to sub-arcsec). Position: $x_v = a(\cos E - e)$, $y_v = a\sqrt{1-e^2}\sin E$, $\nu = \operatorname{atan2}(y_v,x_v)$, $r = \sqrt{x_v^2+y_v^2}$, rotated by $(\omega, \Omega{=}N, i)$ to a heliocentric ECLIPTIC vector (`:93-108`).
- **Assembly**: `batch_solve_ephemeris` (`src/engine/wasm_compute/src/ephemeris.rs:117-256`) — Sun pinned at origin (`:129`), per-body local vectors, recursive parent-vector addition `resolve_absolute` (depth cap 10, unknown parent → zero vector, `:139-170`), geocentric = absolute − Earth (`:186-191`).
- **Magnitude**: $mag = mag_{base} + 5\log_{10}(r_{helio}\cdot dist_{au})$ (`:200`), Sun hardcoded −26.7 (`:197-198`); skip non-Luna moons when mag>20 (`:204`). **No phase-angle term at all.**
- **Ecliptic→equatorial**: FIXED obliquity $\varepsilon = 23.43929°$ (`:179`): $y_{eq} = y\cos\varepsilon - z\sin\varepsilon$, $z_{eq} = y\sin\varepsilon + z\cos\varepsilon$ (`:207-209`); RA converted to HOURS (`:214-215`). Element tables: `src/engine/core/celestial_data.ts:38-97` (Sun→Luna; e.g. Neptune M=260.2471, Moon M=115.3654 — Schlyter's table values verbatim), minor moons `:100-219`.
- **Cite**: element sets & degree-space Newton iteration match Paul Schlyter, "Computing planetary positions" (stjarnhimlen.se/comp/ppcomp.html) [VERIFY — tutorial source, not peer-reviewed]. The distance-modulus magnitude convention ($V(1,0) + 5\log_{10}(r\Delta) + \text{phase}$) is the Astronomical Almanac system — Mallama & Hilton (2018), Astronomy & Computing 25, 10.
- **Gotcha (mixed-convention magBase — fixed)**: the law is only valid with absolute $V(1,0)$-style constants; the DB originally mixed conventions (Jupiter −2.70, Saturn 0.46, Uranus 5.52, Neptune 7.84, Pluto 14.3, Luna −12.74 were mean APPARENT magnitudes — Jupiter at opposition computed ≈ +4.0 vs true ≈ −2.7; the Moon ≈ −25.7). All entries are now $V(1,0)$: Jupiter −9.40, Saturn −8.88 (globe only, no rings), Uranus −7.19, Neptune −6.87, Pluto −0.45 ($H_V$), Luna +0.21 (law then yields the FULL-MOON-equivalent value ≈ −12.7 — phase still not modeled); satellite rows derived via $H = m_{opp} - 5\log_{10}(a_{parent}\cdot(a_{parent}-1))$, which reproduces published H (Phobos 11.8, Titan −1.3, Janus 4.9, …). Consequence of the fix: honest satellite mags are all <20, so the `mag>20` moon skip (`:204`) no longer prunes them — all bodies now appear in results. Regression anchor: `src/engine/tests/ephemeris.test.ts` pins Jupiter ≈ −2.9 (2023-11-03 opposition) and Saturn ≈ +0.5 (2023-08-27) through the REAL compiled WASM. `mag` remains display/routing-grade only (no phase term); do not use it photometrically. Also: `batch_solve_ephemeris` takes a `_jd` argument it IGNORES (`:122`) — d is the only clock. Shipped-binary caveat: this fn runs from the compiled `pkg/` WASM; a stale-binary incident on record means any edit here is invisible until a `wasm-pack` rebuild (N/A for this fix — magBase flows in from TS at call time).

### 5. Epoch-adjusted mean anomaly pre-compensation (Schlyter epoch + parent-mass scaling + Moon rate)
Corrects a subtle mismatch between the Kepler solver's fixed reference epoch and the orbital-element table's own reference epoch, and between the Sun-derived orbital rate and a satellite's true (parent-mass-scaled) rate — without this step every moon's mean anomaly is measurably wrong.
- **Problem**: the WASM always derives n from the SUN's GM (entry 4); for satellites that is wrong by $\sqrt{M_{sun}/M_{parent}}$, and the DB's M values are Schlyter-epoch $d_0 = JD\ 2451543.5$ (1999-12-31 00:00 UTC), 1.5 days before J2000.
- **Formula**: $M' = (M + n_{true}\cdot 1.5 + (n_{true} - n_{wasm})\cdot d) \bmod 360$ (clamped ≥0), constructed so $M' + n_{wasm}\cdot d \equiv M + n_{true}\cdot(d + 1.5)$ at the single requested d. $n_{wasm} = 0.9856076686/a^{1.5}$; $n_{true} = n_{wasm}$ (heliocentric), 13.0649929509 °/day (Luna — Schlyter's anomalistic Moon rate, $\approx 360°/27.55455\ d$), $n_{wasm}\cdot\sqrt{M_{parent}/M_{sun}}$ otherwise ($M_{sun} = 1.989\times10^{30}\ kg$).
- **Code**: `src/engine/core/EphemerisEngine.ts:122-143` (`epochAdjustedMeanAnomaly`; SCHLYTER_EPOCH_OFFSET_DAYS=1.5 at `:127`, $n_{wasm}$ `:130`, $n_{true}$ ladder `:134-138`, $M'$ `:140-142`); injected at `:52` before the WASM batch call `:57`. Rationale comment `:106-121`.
- **Cite**: Kepler's third law $n \propto \sqrt{GM_{parent}}/a^{1.5}$ — textbook two-body mechanics; epoch/rate values from Schlyter [VERIFY — tutorial source].
- **Gotcha**: the compensation is exact only AT the requested d — the adjusted elements are single-instant, not reusable. The +1.5 d epoch shift is $1.5n$ of mean anomaly — ~19.6° for the Moon (the original bug) and NOT sub-degree for all planets: ~6.1° Mercury, ~2.4° Venus, ~1.5° Earth, <1° only from Mars outward; the in-code comment's "Harmless for planets (<1 deg)" (`:119`) understates the inner planets (all fully compensated either way, since heliocentric $n_{true}=n_{wasm}$ still gets the $+1.5n$ term). Non-Luna satellites are honestly labeled "decorative markers" in the comment (`:133`) — their parent-mass rate fixes motion, not their crude i/N/w≈0 elements. Results memoized by (date,lat,lon) (`:21-27`). `FAINT_MOON_MAG_CUTOFF=12` at `:11` is a dead constant (never referenced).

### 6. Topocentric parallax correction (Moon / dist < 0.1 AU)
Shifts a nearby body's apparent position (in practice, only the Moon) to account for the observer not standing at Earth's center — geocentric and topocentric positions of a body this close can differ by more than an arcminute.
- **Trigger**: body_type=="MOON" OR $dist_{au} < 0.1$ (`ephemeris.rs:219-223`).
- **Formula**: $\pi = \arcsin(R_{earth}/dist_{km})$, $R_{earth} = 6378.14\ km$, $AU = 149597870.7\ km$ (`:263-265`); clock = GMST hours-form (entry 2 Form C) → LST → HA in radians (`:267-271`); $A = \cos\delta\sin H$, $B = \cos\delta\cos H - \cos\phi\sin\pi$, $C = \sin\delta - \sin\phi\sin\pi$ (`:278-280`); $H' = \operatorname{atan2}(A,B)$; $\alpha' = LST - H'$ (HOURS, wrapped +24); $\delta' = \operatorname{atan2}(C\cos H', B)$ (`:282-286`).
- **Code**: `src/engine/wasm_compute/src/ephemeris.rs:258-289` (`apply_topocentric_correction_internal`), wasm export `:291-299`.
- **Cite**: HA-space parallax reduction after Meeus, *Astronomical Algorithms*, ch. 40 form [VERIFY — book] — but SIMPLIFIED: $\rho\sin\phi'/\rho\cos\phi'$ are replaced by $\sin\phi/\cos\phi$ (`:275-276`).
- **Gotcha**: **spherical Earth, sea-level observer** — no geodetic→geocentric latitude conversion (Meeus's $\rho$, $\phi'$ with flattening 1/298.257 omitted; worst-case ~0.19° latitude error → arcsec-to-arcmin-scale Moon shift) and no observer elevation. $\pi$ uses asin (fine). Same shipped-binary caveat as entry 4: this function is frozen pending rebuild after any Rust-side edit.

### 7. Timestamp trust ladder + stack exposure-midpoint
Decides how much to trust a frame's embedded timestamp before letting any downstream ephemeris/altitude/airmass computation use it — a bogus clock (dead camera battery, unset RTC) would otherwise silently produce a plausible-looking but wrong planetary overlay.
- **Ladder** (enum `'EXIF' | 'FITS' | 'DEFAULT' | 'USER'`, `src/engine/types/schema.ts:136`): EXIF path prefers DateTimeOriginal > CreateDate > ModifyDate — all stamped `'EXIF'` (`m1_ingestion/metadata_reaper.ts:214-223`); FITS path stamps `'FITS'` (below); `'USER'` only ever set by a manual UI edit (`ui/PipelineWizard.tsx:299`).
- **FITS midpoint** (`m1_ingestion/fits_decoder.ts:331-390`): DATE-OBS forced UTC when no TZ designator (regex `/(?:Z|[+-]\d{2}:?\d{2})$/i`, `:342-346`). If DATE-END exists — or Siril-convention EXPSTART/EXPEND (Julian Dates) with $0 < \text{span} \le 2$ days (`:365-368`; guard exists because real 446-day multi-month composites were observed, `:354-356`) — timestamp = (start+end)/2, the integration midpoint (`:369-373`). Honest degrade: stacked file (STACKCNT>1) with no end marker keeps DATE-OBS as-is + an explicit "may be offset from the integration midpoint" warning rather than guessing a direction (`:374-380`, rationale `:334-341`). No DATE-OBS at all → EXPSTART JD (sanity > 2400000) with warning (`:385-390`). STACKCNT/NSTACK + LIVETIME/TOTALEXP captured either way (`:349-352`).
- **Consumer gate**: `clockReal = i.timestampTrusted && meta?.timestamp_source && meta.timestamp_source !== 'DEFAULT'` (`pipeline/stages/psf_attribution.ts:349`) gates all observing-geometry outputs; `timestampTrusted` itself comes from unset-clock forensics — year < 2004, > now+2 d, or the Jan-1 00:xx dead-battery signature (`getUTCHours()===0` — the whole midnight hour, not just 00:0x) → untrusted (`pipeline/orchestrator_session.ts:304-327`).
- **Cite**: DATE-OBS-is-UTC and DATE-END semantics: Pence, Chiappetti, Page, Shaw & Stobie (2010), "Definition of the Flexible Image Transport System (FITS), version 3.0", A&A 524, A42. EXPSTART/EXPEND-as-JD is a Siril tool convention [VERIFY — software convention, not a standard].
- **Gotcha**: `'DEFAULT'` IS assigned — it is the initialized baseline of every reaper metadata object (`metadata_reaper.ts:39,144,281`: `createDefaultHard` plus the EXIF and FITS paths' initial `hard`, always paired with `timestamp = new Date().toISOString()`) and the UI fallback (`MainApp.tsx:51`); it survives whenever no EXIF/FITS stamp is found. That makes the enum check meaningful rather than decorative: a defaulted timestamp is wall-clock "now", which PASSES the unset-clock forensics (plausible year, not future) and reaches the gate with `timestampTrusted=true` — the explicit `!== 'DEFAULT'` check at `psf_attribution.ts:349` is the only thing failing it closed. The plate solve is deliberately time-independent; only ephemeris/alt-az/airmass consumers ride this ladder.

### 8. Astrometric time inversion — first-order GMST→JD inverse (forensic timestamp recovery)
A one-off design exploration for recovering an unknown capture time from a few anchor-star altitude measurements — inverting the sidereal-time formula given the position of known stars in the frame. It is preserved here as a historical design record; the code has since been removed.
- **Formula**: per anchor star, $\cos H = \dfrac{\sin(alt) - \sin\phi\sin\delta}{\cos\phi\cos\delta}$ (`src-tauri/src/solver/sextant.rs:86`, guards: alt ∈ [1°,85°] `:69`, pole/zenith divisors `:81-83`, $|\cos H| \le 1$ `:89`); east/west sign from pixel_x > crpix[0] (`:101-106`); $LST_h = (LST°/15 + 24) \bmod 24$ (`:109`); $GMST_h = (LST_h - lon/15 + 24) \bmod 24$ (`:112`); then $T \approx (GMST_h - 6.697374558)/2400.051336$, $JD = 2451545.0 + 36525\cdot T$ (`:153-156`). Consensus = median of per-anchor JDs (`:122-129`) + std-dev spread (`:131-135`).
- **Code**: `src-tauri/src/solver/sextant.rs:49-156` (`solve_with_dem` + `gmst_to_jd`) — **FILE DELETED @7288cfc; line refs are to the historical blob at deletion time.** NOTE: this Rust-side lane worked in RA **DEGREES** (`StarAltitudeAnchor.ra_deg`, `:13-15`) — opposite of the TS-side hours convention.
- **Cite**: inverts the 0h-UT GMST seed polynomial $GMST = 6.697374558 + 2400.051336\cdot T + 0.000025862\cdot T^2$ (comment `:147-151`) — Meeus, *Astronomical Algorithms*, eq. 12.2 lineage [VERIFY — book], first-order ($T^2$ term dropped).
- **Gotcha**: **deleted — section preserved as the design record for a digital-sextant P1 feature, not live code** (at deletion: compiled and unit-tested, no production caller; only `pub mod sextant;` in `solver/mod.rs:2` plus its own `#[cfg(test)]` block). And the inversion as written is **date-ambiguous by construction**: GMST is reduced mod 24 h before inversion, so $T \in [-0.0028, +0.0072)$ centuries — every recovered JD lands within ~1 year of J2000 regardless of the true capture epoch. It recovers sidereal PHASE (time of day), not a date; a caller would have to supply the day number externally. The east/west disambiguation via pixel-x vs CRPIX assumes a specific camera orientation/parity — unverified proxy.

## UNIT WARNINGS
- RA is **HOURS** on the TS side everywhere (TimeService args, wasm ephemeris results `ra`, `planetary_adapter` `ra_hours`) — but **DEGREES** on the Rust solver side (`sextant.rs` `ra_deg`) and in FITS headers (`RA` header → `ra_hint` via /15, `fits_decoder.ts:401-403`).
- GMST/LST carry THREE numeric conventions (entry 2): degrees-full (TimeService), degrees-truncated (psf_physics, image_conditions), hours-linear (all WASM + planetary_adapter). Mixing an hours value into a degrees formula is silent — no runtime unit check anywhere.
- Kepler math (entry 4) is in **DEGREES** with explicit $180/\pi$ factors inside the Newton step — porting it to a radians library without dropping those factors double-converts.
- Orbital-element epoch: CELESTIAL_DB mean anomalies are **Schlyter $d_0 = JD\ 2451543.5$**, NOT J2000 — raw elements must pass through `epochAdjustedMeanAnomaly` (entry 5) before the WASM; feeding them directly loses 1.5 days (≈19.6° for the Moon).
- `batch_solve_ephemeris(bodies, lat, lon, d, _jd)` ignores its jd argument — d (days since J2000) is the only clock; passing jd where d is expected shifts the epoch ~2.45 million days (≈6,700 years) into the future.
- Time scale is UTC-as-UT1 throughout — no ΔT/leap-second/TDB handling in any entry above.
- Catalog reference frame is separate from all of the above: Gaia DR3 star positions (the atlas this codebase solves against) are given at reference epoch J2016.0, not J2000 — see CARD_STARPLATES_DATAPLANE.md #3 for how (and where) that gap is, and is not, accounted for.
