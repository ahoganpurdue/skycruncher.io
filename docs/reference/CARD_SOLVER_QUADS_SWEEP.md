### 1. Canonical 4D quad code (farthest-pair similarity frame)
- **Formula**: from the 4 points selected by `idx`, pick (A,B) = argmax over the 6 pairs of |P_m−P_n|². Frame: u=B−A, det=|u|²=ux²+uy² (reject det<1e-9). T(p) = ( ((p−A)·u)/det, (−dx·uy+dy·ux)/det ) — maps A→(0,0), B→(1,0). Code = (x_C, y_C, x_D, y_D) with C,D the two non-axis points. Mirror normalization: if x_C+x_D > 1, apply (x,y)→(1−x,−y) to both C and D and swap A↔B. Ordering: if x_C > x_D (or x_C = x_D and y_C > y_D) swap C↔D. Compactness reject: all 4 components must be finite with |v| ≤ 4.0.
- **Code**: `src/engine/wasm_compute/src/solver_planar.rs:11-88` (`build_quad`; farthest pair `:18-32`, transform `:58-62`, det reject `:56`, mirror norm `:67-71`, lex order `:73-76`, |v|≤4.0 reject `:78-82`). The idx-selection bug is documented IN-FILE at `:12-16`: the original code read `p[0..4]` directly instead of `p[idx[..]]`, so every combination hashed the array's first four points — all quads in one bin, no genuine match possible. Fixed at `:16` (`pts = [p[idx[0]], …]`).
- **Cite**: Lang, Hogg, Mierle, Blanton & Roweis (2010), AJ 139, 1782, §2 — same A/B-swap (x_C+x_D≤1) and C/D-swap (x_C≤x_D) symmetry breaking. Deviations: their canonical frame puts B at (1,1) (45°-rotated frame), this code puts B at (1,0); they require C,D inside the circle with AB as diameter, this code accepts anything in the ±4.0 box.
- **Gotcha**: the returned `indices` are remapped through BOTH swaps (`:85`), so index order always means (A,B,C,D) in the canonical frame — consumers must not assume input order. The ±4.0 box is far looser than astrometry.net's in-circle constraint: quads whose C/D lie well outside the A-B span still hash, they just spread across more bins.

### 2. Quad hash matching (floor bins + neighbor walk + mirrored-twin dual pass)
- **Formula**: bins_i = floor(code_i / 0.05) (bin width 0.05, `:91`). Probe set per pass = bins + (dx,dy,dz,dw), each offset ∈ {−1,0,+1} — the full 3⁴ = 81-bin hypercube, not just axis probes (`:191-195`). Mirrored twin: a parity-flipped frame negates the y components, so a second pass runs with (c₀,−c₁,c₂,−c₃) whenever its bins differ from the direct bins (`:181-188`). Accept when Σᵢ(code_used_i − cat_i)² < admit; admit starts at tol² (`:171`) and tightens to the worst kept distance once the buffer fills (MAX_KEPT=2000 → sort, truncate to 1000, `:166,210-214`). Tolerance ladder (TS side): default [0.01, 0.02, 0.03] for `quadtolerance` = `SOLVER_QUAD_tolerance_DEFAULT` = 0.01 (`pipeline_config.ts:68`; wider ladders for larger tolerances, `solver_entry.ts:1399-1408`); first non-empty tolerance wins (`break`, `:231`), best FINAL_TAKE=200 by distance returned as 9-double rows (4 det ids, 4 cat ids, dist², `:205-209`).
- **Code**: `src/engine/wasm_compute/src/solver_planar.rs:90-98` (`get_hash_bins`), `:100-236` (`solve_planar_local`; quad pools `:128-153`, dual-pass parity fix documented in-file `:174-188`, bin walk `:190-199`, L2 `:201-202`, bounded top-K `:162-215`). TS call site: `solver_entry.ts:1465-1471`, `max_stars` = max(`SOLVER_MAX_DET_STARS`=40, `SOLVER_MAX_CAT_STARS`=100) (`pipeline_config.ts:65-66`).
- **Cite**: Lang et al. (2010) for the code space; the retrieval structure deviates — they use kd-tree nearest-neighbor search over codes, this is a house floor-bin hash with an 81-bin walk. The mirrored-twin walk has no counterpart in the paper (astrometry.net indexes both parities instead).
- **Gotcha**: the in-file comment `:174-180` documents the fixed bug — the old code computed a mirror distance but only walked the UNmirrored bins, so parity-flipped (e.g. FITS bottom-up) frames could never match. The mirrored twin is NOT re-canonicalized after y-negation (x components are untouched so the C/D order survives except exact x-ties — measure-zero). Separately, `pipeline_config.ts:103`'s "hash bins are ~1%" conflates the L2 tolerance (0.01–0.03) with the bin width (0.05): the walk covers ±0.05–0.10 per axis; the ~1% figure is the ACCEPT distance, which is what actually rejects distorted codes.

### 3. Anchored rotation sweep (translation pinned, θ×parity brute force, self-null z-gate)
- **Formula**: translation pinned to a bright-anchor hypothesis (crpix := anchor pixel, crval := center under test), scale pinned by metrology. Score m(θ,p) = #{catalog stars with mag_V < 6.0 whose rotated projection px = aCx + (ξ·cosθ − p·η·sinθ)/degPerPx, py = aCy + (ξ·sinθ + p·η·cosθ)/degPerPx lands in-frame with a detection within tol}, tol = max(baseNetPx, 0.035·r_offaxis), baseNetPx = min(2000, max(120, 15·pixelScale))/pixelScale px; degPerPx = pixelScale/3600. Grid: θ ∈ {0, 0.5, …, 359.5} × p ∈ {+1,−1} = **1440 orientations**. Null = mean/σ of those same 1440 scores (peak included); sweep-z = (m_peak − mean)/σ, σ = √max(1e-9, E[m²]−mean²) ∥ 1. Gate: z ≥ `SOLVER_UW_SWEEP_MIN_Z` = 4.5 → fine refine θ±1.5° @ 0.1° (same parity) → CD built directly in the sweep's own convention, CD = degPerPx·[[cosθ, sinθ], [−p·sinθ, p·cosθ]] (DEG/px) → full `verifyWCS`. Sub-threshold peaks in [`SOLVER_UW_ESCALATE_MIN_Z`=3.0, 4.5) route to forced-photometry escalation instead of dying.
- **Code**: `src/engine/pipeline/m6_plate_solve/solver_entry.ts:982-1160` (block comment `:982-995`, arming condition `:996-999` — ultra-wide patch + anchor mode + `planar_local` strategy (the default) + ≥12 bright catalog stars; detection lookup grid, cell 128 px, `:1000-1014`; net `:1015-1016`; `scoreOrientation` `:1035-1048`; θ×p loop `:1081-1088`; null stats `:1089-1091`; gate `:1106`; refine `:1134-1138`; CD + crval=[ra0 HOURS, dec0 DEG] `:1140-1153`; verify `:1154`; escalation branch `:1162` on). Constants + calibration rationale: `pipeline_config.ts:126-134` (mag limit 6.0 `:126`, "true orientation peaks at z=+7.1; best junk +2.8" `:133`, gate 4.5 `:134`), escalation floor `:205`.
- **Cite**: house method — no direct literature lineage for the sweep itself. The 4.5σ gate placement is justified in docs (NOT in code — only the calibrated constant ships) by the classical expected-maximum bound for M iid N(0,1) draws, E[max] ≈ √(2 ln M): Fisher & Tippett (1928), Proc. Camb. Phil. Soc. 24, 180; Gumbel, *Statistics of Extremes* (1958). Theorem application: `docs/archive/SOLVER_TOOLSET.md:257-289` (§2.5) and `docs/01-canonical/processing_flow.md:97` (repo root).
- **Gotcha**: DOC-VS-CODE DRIFT (historical, preserved here as `SOLVER_TOOLSET.md` was archived 2026-07-10 → `docs/archive/SOLVER_TOOLSET.md`) — that doc once computed the noise ceiling with "M≈720 (θ×parity)" → √(2 ln 720) ≈ 3.63σ, but the code evaluates 720 θ × 2 parities = **1440** orientations (which `pipeline_config.ts:131` states correctly); the honest ceiling is √(2 ln 1440) ≈ 3.81σ. The doc has since been self-corrected to M≈1440 (`docs/archive/SOLVER_TOOLSET.md:261,267`), so the M-halving is now resolved; the stale `solver_entry.ts:771` citation at its `:260` persists (the sweep lives at `:982-1160` now). Statistically: the null includes the peak and its correlated neighbor lobes (scores are NOT iid), so sweep-z is an indicative score, not a calibrated p-value — §2.5's EVT correction is acknowledged future work, not shipped.

### 4. Winner-dominance margin (lever centers only)
- **Formula**: runner-up = max{s.m : NOT(s.parity = p_peak AND dθ_circ(s.θ, θ_peak) < sepDeg)}, where dθ_circ = |Δθ| mod 360 folded to ≤180 (`:166-167`); if no eligible score, runner-up := nullMean (peak stands alone, `:172`). z_x = (m_x − nullMean)/(nullStd ∥ 1). dominant ⇔ (z_peak − z_runnerUp) ≥ marginZ. Live values: sepDeg = `SOLVER_UW_DOMINANCE_MIN_SEP_DEG` = 20, marginZ = `SOLVER_UW_DOMINANCE_MARGIN_Z` = 1.5 (`pipeline_config.ts:186-187`).
- **Code**: `src/engine/pipeline/m6_plate_solve/fine_center_lever.ts:140-176` (`orientationDominance`; rationale comment `:140-153`). Wiring: `solver_entry.ts:1106-1132` — applies ONLY when the center carries `lever:true` and only AFTER the base 4.5σ gate passes (evidence-add, never a gate change); per-orientation scores are retained solely for lever centers (`:1078,1085` — pushes are `leverCenter`-gated; the empty array itself is allocated either way, contra the in-code "no alloc" comment at `:1077`).
- **Cite**: house heuristic — closest literature analogue is the best-vs-second-best ratio test of Lowe (2004), "Distinctive Image Features from Scale-Invariant Keypoints", IJCV 60(2), 91 [VERIFY — analogy only, not a lineage; this code uses a σ-difference margin on an anchored-rotation score, not a descriptor-distance ratio].
- **Gotcha**: the lobe exclusion is SAME-PARITY only — an opposite-parity score at ANY angle (including θ_peak itself) is an eligible runner-up, so a mirror-ambiguous field correctly fails dominance. Failure handling is asymmetric (`:1127-1130`): anchor#0 non-dominant aborts the whole center (`return {success:false}` — preserves the pre-§2a behavior byte-identically), while alternative anchors #1/#2 just yield to the next candidate.

### 5. Fine-center lever geometry (anchor dedupe + center-out grid)
- **Formula**: great-circle sep(1,2) = acos(clamp(sin δ₁ sin δ₂ + cos δ₁ cos δ₂ cos(α₁−α₂), ±1)) with α = ra_hours·15·π/180 (`:38-44`). Dedupe: sort anchors by `priority` ascending (planets 0, injected priors 1, bundled bright stars 2+i/1000, trusted hint −1), keep an anchor unless sep < dedupDeg from an already-kept one (`:52-61`). Grid per anchor: Δdec, Δra_deg ∈ {−halfDeg … +halfDeg} step stepDeg (loop bound +1e-9 so the endpoint survives float accumulation), ra = anchor.ra + (Δra_deg/15)/cos(δ) (degrees→HOURS then sky-uniform 1/cos δ stretch, `:75`; pole guard cos δ ∥ 1e-6, `:70`), sorted center-out by r = hypot(Δra_deg, Δdec) (`:82`). Budget: total output capped at maxCenters ACROSS all anchors (`:100-107`). Live values: halfDeg=1.5, stepDeg=0.3, maxCenters=400, dedupDeg=1.0 (`pipeline_config.ts:175-178`) → 11×11 = 121 centers/anchor, so the cap funds ~3⅓ anchors.
- **Code**: `src/engine/pipeline/m6_plate_solve/fine_center_lever.ts:38-44` (`angularSepDeg`), `:46-61` (`dedupeAnchors`), `:63-85` (`fineGridAround`), `:87-109` (`buildFineCenters`). Two call sites in `solver_entry.ts`: `:365-379` (A2 — trusted-hint neighborhood, PREPENDED, ultra-wide + trusted hint only) and `:466-497` (anchor lever — APPENDED after the coarse sweep as a fallback, ultra-wide only; both FOV-gated by `SOLVER_WIDE_PATCH_MIN_FOV_DEG`=30 so narrow SeeStar frames are byte-identical).
- **Cite**: — (no literature lineage: elementary spherical geometry + greedy dedupe; the RA/cos δ spacing is the standard true-angle correction, textbook-grade, e.g. Smart, *Textbook on Spherical Astronomy*, 6th ed. 1977 [VERIFY — general reference, not a specific algorithm source]).
- **Gotcha**: the acos-form separation is ill-conditioned below ~1e-4° — fine at the 1.0° dedupe radius, do NOT reuse it for sub-arcsecond work. The pole guard (cos δ ∥ 1e-6) prevents Infinity but makes RA offsets explode as |δ|→90° (grid degenerates toward a meridian sliver). No RA wrap normalization — output `ra` can leave [0,24) (harmless: every consumer is trigonometric). Rationale figures in the file header (`:6-8`, sweep fires within ~0.5°; 0.8° error collapsed the true peak 8.5σ→2.4σ) match `pipeline_config.ts:166-171` — no drift.

## UNIT WARNINGS
- RA is **HOURS** throughout this lane (`AnchorCenter.ra`, `FineCenter.ra`, `crval[0]`, `angularSepDeg` args — it multiplies by 15 internally, `fine_center_lever.ts:40`). Grid RA offsets are computed in DEGREES then divided by 15 (`:75`) — reorder those two ops and everything silently lands 15× off.
- Quad codes and quad tolerances are **DIMENSIONLESS** similarity-frame ratios. Both pools must be in the SAME pixel units before quad building: catalog ξ,η (DEGREES, TS `gnomonicProject`) are converted at `solver_entry.ts:961-979` — primary path ScaleManager `degreeToScaledCanvas`, fallback ÷ degPerPx — **both with a y-flip** (y = origin − scaled η); the sweep instead leaves y unflipped and lets the parity variable absorb mirroring — two different conventions in the same file.
- TS-side `WCSTransform.cd` is **DEG/px** (sweep CD built with degPerPx = pixelScale/3600, `solver_entry.ts:1145-1153`); the Rust `sky_transform.rs` fit works in rad/px (see the WCS card) — conversion happens at the WASM boundary (`SkyTransform.ts:115,253`). Mixing the two is silent.
- Sweep θ is DEGREES (converted to radians per call, `solver_entry.ts:1036`); dominance angular distance is degrees on the 360° circle (`fine_center_lever.ts:166-167`).
- Three different σ scales coexist here: detection SNR σ, verify-stack z, and **sweep-z** (units of the sweep's own 1440-score σ). `SOLVER_UW_SWEEP_MIN_Z`, `SOLVER_UW_ESCALATE_MIN_Z`, and `SOLVER_UW_DOMINANCE_MARGIN_Z` are all in sweep-σ; they are not comparable to verify or photometry σ values.
