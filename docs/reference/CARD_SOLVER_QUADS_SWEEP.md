# Quad Codes & Rotation Sweep — algorithm card (solver internals beyond the WCS card)

This card goes one level deeper than CARD_ASTROMETRY_WCS.md into the legacy blind solver: exactly how a 4-star quad gets turned into a lookup code, how that code is matched against the catalog index, and — for wide fields where quad matching alone isn't reliable — how a brute-force rotation sweep anchored on a bright star finds the true orientation instead.

Symbols: (x,y)=pixel; code=(x_C,y_C,x_D,y_D)=dimensionless 4D similarity-frame quad code; θ=sweep rotation angle, DEGREES; p=parity ∈{+1,−1}; (ξ,η)=tangent-plane coords in DEGREES (TS side — the Rust `sky_transform.rs` radians convention is the WCS card's lane); (α,δ)=RA/Dec, **RA in HOURS** unless stated; z=sweep-score deviate in units of the sweep's OWN score σ.
Every entry below is VERIFIED against the code at the cited line; formulas transcribed from the running code, not idealized.

> **Cold-path note (legacy vs. current reference engine):** every entry on this card — the canonical quad code (entry 1), quad-hash matching (entry 2), anchored rotation sweep (entry 3), winner-dominance margin (entry 4), and fine-center lever geometry (entry 5), plus the UNIT WARNINGS below — documents the legacy TS/WASM solver control flow (`solver_entry.ts`, `wasm_compute/src/solver_planar.rs`, `fine_center_lever.ts`). The shipping reference solver is now a separate Rust core on `crates/solver-core` — a different control loop (abort-on-accept + band-major search; no anchor/rotation sweep, no batch generate-cluster-judge, none of the `SOLVER_UW_*` sweep-tuning flags cited below apply to it). See `test_results/greenfield_solver/NIGHT_LEDGER_2026-07-21.md` for the cutover record. The formulas below remain an accurate description of the retained legacy code path, not the current core's algorithm.

### 1. Canonical 4D quad code (farthest-pair similarity frame)
Turns any 4 stars into a single, rotation- and scale-invariant 4-number fingerprint by picking the two farthest-apart stars as a local coordinate frame and expressing the other two relative to it — this is what makes quad matching independent of a field's orientation and plate scale.
- **Formula**: from the 4 points selected by `idx`, pick $(A,B) = \arg\max$ over the 6 pairs of $|P_m-P_n|^2$. Frame: $u=B-A$, $\det=|u|^2=u_x^2+u_y^2$ (reject $\det<10^{-9}$). $T(p) = \left(\dfrac{(p-A)\cdot u}{\det},\ \dfrac{-dx\,u_y+dy\,u_x}{\det}\right)$ — maps $A\to(0,0)$, $B\to(1,0)$. Code = $(x_C, y_C, x_D, y_D)$ with C,D the two non-axis points. Mirror normalization: if $x_C+x_D > 1$, apply $(x,y)\to(1-x,-y)$ to both C and D and swap A↔B. Ordering: if $x_C > x_D$ (or $x_C = x_D$ and $y_C > y_D$) swap C↔D. Compactness reject: all 4 components must be finite with $|v| \le 4.0$.
- **Code**: `src/engine/wasm_compute/src/solver_planar.rs:11-88` (`build_quad`; farthest pair `:18-32`, transform `:58-62`, det reject `:56`, mirror norm `:67-71`, lex order `:73-76`, |v|≤4.0 reject `:78-82`). The idx-selection bug is documented IN-FILE at `:12-16`: the original code read `p[0..4]` directly instead of `p[idx[..]]`, so every combination hashed the array's first four points — all quads in one bin, no genuine match possible. Fixed at `:16` (`pts = [p[idx[0]], …]`).
- **Cite**: Lang, Hogg, Mierle, Blanton & Roweis (2010), AJ 139, 1782, §2 — same A/B-swap ($x_C+x_D\le1$) and C/D-swap ($x_C\le x_D$) symmetry breaking. Deviations: their canonical frame puts B at (1,1) (45°-rotated frame), this code puts B at (1,0); they require C,D inside the circle with AB as diameter, this code accepts anything in the ±4.0 box.
- **Gotcha**: the returned `indices` are remapped through BOTH swaps (`:85`), so index order always means (A,B,C,D) in the canonical frame — consumers must not assume input order. The ±4.0 box is far looser than astrometry.net's in-circle constraint: quads whose C/D lie well outside the A-B span still hash, they just spread across more bins.

### 2. Quad hash matching (floor bins + neighbor walk + mirrored-twin dual pass)
Looks up a quad's code in the catalog index by floor-dividing it into a coarse grid of bins and walking the neighboring bins, and runs a second, mirror-flipped pass so that flipped (e.g. bottom-up) images can still match.
- **Formula**: $bins_i = \lfloor code_i / 0.05 \rfloor$ (bin width 0.05, `:91`). Probe set per pass = bins + (dx,dy,dz,dw), each offset $\in \{-1,0,+1\}$ — the full $3^4 = 81$-bin hypercube, not just axis probes (`:191-195`). Mirrored twin: a parity-flipped frame negates the y components, so a second pass runs with $(c_0,-c_1,c_2,-c_3)$ whenever its bins differ from the direct bins (`:181-188`). Accept when $\sum_i(code_{used,i} - cat_i)^2 < admit$; admit starts at $tol^2$ (`:171`) and tightens to the worst kept distance once the buffer fills (MAX_KEPT=2000 → sort, truncate to 1000, `:166,210-214`). Tolerance ladder (TS side): default [0.01, 0.02, 0.03] for `quadtolerance` = `SOLVER_QUAD_tolerance_DEFAULT` = 0.01 (`pipeline_config.ts:68`; wider ladders for larger tolerances, `solver_entry.ts:1399-1408`); first non-empty tolerance wins (`break`, `:231`), best FINAL_TAKE=200 by distance returned as 9-double rows (4 det ids, 4 cat ids, dist², `:205-209`).
- **Code**: `src/engine/wasm_compute/src/solver_planar.rs:90-98` (`get_hash_bins`), `:100-236` (`solve_planar_local`; quad pools `:128-153`, dual-pass parity fix documented in-file `:174-188`, bin walk `:190-199`, L2 `:201-202`, bounded top-K `:162-215`). TS call site: `solver_entry.ts:1465-1471`, `max_stars` = max(`SOLVER_MAX_DET_STARS`=40, `SOLVER_MAX_CAT_STARS`=100) (`pipeline_config.ts:65-66`).
- **Cite**: Lang et al. (2010) for the code space; the retrieval structure deviates — they use kd-tree nearest-neighbor search over codes, this is a house floor-bin hash with an 81-bin walk. The mirrored-twin walk has no counterpart in the paper (astrometry.net indexes both parities instead).
- **Gotcha**: the in-file comment `:174-180` documents the fixed bug — the old code computed a mirror distance but only walked the UNmirrored bins, so parity-flipped (e.g. FITS bottom-up) frames could never match. The mirrored twin is NOT re-canonicalized after y-negation (x components are untouched so the C/D order survives except exact x-ties — measure-zero). Separately, `pipeline_config.ts:103`'s "hash bins are ~1%" conflates the L2 tolerance (0.01–0.03) with the bin width (0.05): the walk covers ±0.05–0.10 per axis; the ~1% figure is the ACCEPT distance, which is what actually rejects distorted codes.

### 3. Anchored rotation sweep (translation pinned, θ×parity brute force, self-null z-gate)
For wide fields where the quad hash alone can be unreliable, this pins the frame's translation and scale using a bright anchor star, then brute-force sweeps every rotation angle and parity looking for the orientation that best matches known bright catalog stars against detections — scored against its own self-generated noise floor.
- **Formula**: translation pinned to a bright-anchor hypothesis (crpix := anchor pixel, crval := center under test), scale pinned by metrology. Score $m(\theta,p) = \#\{$catalog stars with $mag_V < 6.0$ whose rotated projection $p_x = aC_x + (\xi\cos\theta - p\,\eta\sin\theta)/degPerPx$, $p_y = aC_y + (\xi\sin\theta + p\,\eta\cos\theta)/degPerPx$ lands in-frame with a detection within tol$\}$, $tol = \max(baseNetPx, 0.035\cdot r_{offaxis})$, $baseNetPx = \min(2000, \max(120, 15\cdot pixelScale))/pixelScale$ px; $degPerPx = pixelScale/3600$. Grid: $\theta \in \{0, 0.5, \ldots, 359.5\} \times p \in \{+1,-1\}$ = **1440 orientations**. Null = mean/σ of those same 1440 scores (peak included); sweep-z = $(m_{peak} - mean)/\sigma$, $\sigma = \sqrt{\max(10^{-9}, E[m^2]-mean^2)} \vee 1$. Gate: $z \ge$ `SOLVER_UW_SWEEP_MIN_Z` = 4.5 → fine refine θ±1.5° @ 0.1° (same parity) → CD built directly in the sweep's own convention, $CD = degPerPx\cdot\begin{bmatrix}\cos\theta & \sin\theta\\ -p\sin\theta & p\cos\theta\end{bmatrix}$ (DEG/px) → full `verifyWCS`. Sub-threshold peaks in [`SOLVER_UW_ESCALATE_MIN_Z`=3.0, 4.5) route to forced-photometry escalation instead of dying.
- **Code**: `src/engine/pipeline/m6_plate_solve/solver_entry.ts:982-1160` (block comment `:982-995`, arming condition `:996-999` — ultra-wide patch + anchor mode + `planar_local` strategy (the default) + ≥12 bright catalog stars; detection lookup grid, cell 128 px, `:1000-1014`; net `:1015-1016`; `scoreOrientation` `:1035-1048`; θ×p loop `:1081-1088`; null stats `:1089-1091`; gate `:1106`; refine `:1134-1138`; CD + crval=[ra0 HOURS, dec0 DEG] `:1140-1153`; verify `:1154`; escalation branch `:1162` on). Constants + calibration rationale: `pipeline_config.ts:126-134` (mag limit 6.0 `:126`, "true orientation peaks at z=+7.1; best junk +2.8" `:133`, gate 4.5 `:134`), escalation floor `:205`.
- **Cite**: house method — no direct literature lineage for the sweep itself. The 4.5σ gate placement is justified in docs (not in code — only the calibrated constant ships) by the classical expected-maximum bound for M iid $N(0,1)$ draws, $E[\max] \approx \sqrt{2\ln M}$: Fisher & Tippett (1928), Proc. Camb. Phil. Soc. 24, 180; Gumbel, *Statistics of Extremes* (1958). Theorem application: `docs/archive/SOLVER_TOOLSET.md:257-289` (§2.5) and `docs/01-canonical/processing_flow.md:97` (repo root).
- **Gotcha**: doc-vs-code drift, since resolved — an archived design doc (`docs/archive/SOLVER_TOOLSET.md`) once computed the noise ceiling with "M≈720 (θ×parity)" → $\sqrt{2\ln 720} \approx 3.63\sigma$, but the code evaluates $720\theta \times 2\ \text{parities} = \mathbf{1440}$ orientations (which `pipeline_config.ts:131` states correctly); the honest ceiling is $\sqrt{2\ln 1440} \approx 3.81\sigma$. The archived doc has since been self-corrected to M≈1440, so the M-halving is resolved. Statistically: the null includes the peak and its correlated neighbor lobes (scores are NOT iid), so sweep-z is an indicative score, not a calibrated p-value — the extreme-value correction that would fix this is acknowledged future work, not shipped.

### 4. Winner-dominance margin (lever centers only)
An extra check applied only to fine-grid "lever" centers near a bright anchor: the winning orientation must not just beat the noise floor, it must clearly beat its next-best same-parity rival, or the candidate is rejected as ambiguous.
- **Formula**: runner-up = $\max\{s.m : \text{NOT}(s.parity = p_{peak}\ \text{AND}\ d\theta_{circ}(s.\theta,\theta_{peak}) < sepDeg)\}$, where $d\theta_{circ} = |\Delta\theta| \bmod 360$ folded to ≤180 (`:166-167`); if no eligible score, runner-up := nullMean (peak stands alone, `:172`). $z_x = (m_x - nullMean)/(nullStd \vee 1)$. dominant ⇔ $(z_{peak} - z_{runnerUp}) \ge marginZ$. Live values: sepDeg = `SOLVER_UW_DOMINANCE_MIN_SEP_DEG` = 20, marginZ = `SOLVER_UW_DOMINANCE_MARGIN_Z` = 1.5 (`pipeline_config.ts:186-187`).
- **Code**: `src/engine/pipeline/m6_plate_solve/fine_center_lever.ts:140-176` (`orientationDominance`; rationale comment `:140-153`). Wiring: `solver_entry.ts:1106-1132` — applies ONLY when the center carries `lever:true` and only AFTER the base 4.5σ gate passes (evidence-add, never a gate change); per-orientation scores are retained solely for lever centers (`:1078,1085` — pushes are `leverCenter`-gated; the empty array itself is allocated either way, contra the in-code "no alloc" comment at `:1077`).
- **Cite**: house heuristic — closest literature analogue is the best-vs-second-best ratio test of Lowe (2004), "Distinctive Image Features from Scale-Invariant Keypoints", IJCV 60(2), 91 [VERIFY — analogy only, not a lineage; this code uses a σ-difference margin on an anchored-rotation score, not a descriptor-distance ratio].
- **Gotcha**: the lobe exclusion is SAME-PARITY only — an opposite-parity score at ANY angle (including θ_peak itself) is an eligible runner-up, so a mirror-ambiguous field correctly fails dominance. Failure handling is asymmetric (`:1127-1130`): anchor#0 non-dominant aborts the whole center (`return {success:false}` — preserves prior behavior byte-identically), while alternative anchors #1/#2 just yield to the next candidate.

### 5. Fine-center lever geometry (anchor dedupe + center-out grid)
Builds the fallback grid of candidate frame centers around each bright "anchor" star (planets, injected hints, bundled bright stars), deduplicated by angular separation and ordered from the anchor outward, that feeds the rotation sweep above when the coarse solve alone isn't enough.
- **Formula**: great-circle $sep(1,2) = \arccos(\operatorname{clamp}(\sin\delta_1\sin\delta_2 + \cos\delta_1\cos\delta_2\cos(\alpha_1-\alpha_2), \pm1))$ with $\alpha = ra_{hours}\cdot15\cdot\pi/180$ (`:38-44`). Dedupe: sort anchors by `priority` ascending (planets 0, injected priors 1, bundled bright stars 2+i/1000, trusted hint −1), keep an anchor unless sep < dedupDeg from an already-kept one (`:52-61`). Grid per anchor: $\Delta dec, \Delta ra_{deg} \in \{-halfDeg \ldots +halfDeg\}$ step stepDeg (loop bound +1e-9 so the endpoint survives float accumulation), $ra = anchor.ra + (\Delta ra_{deg}/15)/\cos\delta$ (degrees→HOURS then sky-uniform $1/\cos\delta$ stretch, `:75`; pole guard $\cos\delta \vee 10^{-6}$, `:70`), sorted center-out by $r = \operatorname{hypot}(\Delta ra_{deg}, \Delta dec)$ (`:82`). Budget: total output capped at maxCenters ACROSS all anchors (`:100-107`). Live values: halfDeg=1.5, stepDeg=0.3, maxCenters=400, dedupDeg=1.0 (`pipeline_config.ts:175-178`) → 11×11 = 121 centers/anchor, so the cap funds ~3⅓ anchors.
- **Code**: `src/engine/pipeline/m6_plate_solve/fine_center_lever.ts:38-44` (`angularSepDeg`), `:46-61` (`dedupeAnchors`), `:63-85` (`fineGridAround`), `:87-109` (`buildFineCenters`). Two call sites in `solver_entry.ts`: `:365-379` (A2 — trusted-hint neighborhood, PREPENDED, ultra-wide + trusted hint only) and `:466-497` (anchor lever — APPENDED after the coarse sweep as a fallback, ultra-wide only; both FOV-gated by `SOLVER_WIDE_PATCH_MIN_FOV_DEG`=30 so narrow SeeStar frames are byte-identical).
- **Cite**: — (no literature lineage: elementary spherical geometry + greedy dedupe; the RA/cos δ spacing is the standard true-angle correction, textbook-grade, e.g. Smart, *Textbook on Spherical Astronomy*, 6th ed. 1977 [VERIFY — general reference, not a specific algorithm source]).
- **Gotcha**: the acos-form separation is ill-conditioned below ~1e-4° — fine at the 1.0° dedupe radius, do NOT reuse it for sub-arcsecond work. The pole guard ($\cos\delta \vee 10^{-6}$) prevents Infinity but makes RA offsets explode as $|\delta|\to90°$ (grid degenerates toward a meridian sliver). No RA wrap normalization — output `ra` can leave [0,24) (harmless: every consumer is trigonometric). Rationale figures in the file header (`:6-8`, sweep fires within ~0.5°; 0.8° error collapsed the true peak 8.5σ→2.4σ) match `pipeline_config.ts:166-171` — no drift.

## UNIT WARNINGS
- RA is **HOURS** throughout this lane (`AnchorCenter.ra`, `FineCenter.ra`, `crval[0]`, `angularSepDeg` args — it multiplies by 15 internally, `fine_center_lever.ts:40`). Grid RA offsets are computed in DEGREES then divided by 15 (`:75`) — reorder those two ops and everything silently lands 15× off.
- Quad codes and quad tolerances are **DIMENSIONLESS** similarity-frame ratios. Both pools must be in the SAME pixel units before quad building: catalog $\xi,\eta$ (DEGREES, TS `gnomonicProject`) are converted at `solver_entry.ts:961-979` — primary path ScaleManager `degreeToScaledCanvas`, fallback ÷ degPerPx — **both with a y-flip** (y = origin − scaled η); the sweep instead leaves y unflipped and lets the parity variable absorb mirroring — two different conventions in the same file.
- TS-side `WCSTransform.cd` is **DEG/px** (sweep CD built with degPerPx = pixelScale/3600, `solver_entry.ts:1145-1153`); the Rust `sky_transform.rs` fit works in rad/px (see the WCS card) — conversion happens at the WASM boundary (`SkyTransform.ts:115,253`). Mixing the two is silent.
- Sweep θ is DEGREES (converted to radians per call, `solver_entry.ts:1036`); dominance angular distance is degrees on the 360° circle (`fine_center_lever.ts:166-167`).
- Three different σ scales coexist here: detection SNR σ, verify-stack z, and **sweep-z** (units of the sweep's own 1440-score σ). `SOLVER_UW_SWEEP_MIN_Z`, `SOLVER_UW_ESCALATE_MIN_Z`, and `SOLVER_UW_DOMINANCE_MARGIN_Z` are all in sweep-σ; they are not comparable to verify or photometry σ values.
